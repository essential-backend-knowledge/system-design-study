# 키-값 저장소 (Key-Value Store) 요구사항 정의서

> **문서 목적**: 분산 키-값 저장소 시스템 설계를 위한 기능적/비기능적 요구사항 정리  
> **작성 기준**: 시스템 규모 — 대규모 트래픽 및 데이터를 처리하는 분산 환경 가정

---

## 1. 개요

키-값 저장소(Key-Value Store)는 고유한 키(Key)와 그에 대응하는 값(Value)을 쌍으로 저장하는 비관계형(Non-relational) 데이터베이스이다.

ex) Redis, Memcached, DynamoDB, etcd 등

### 1.1 핵심 개념 정의

| 용어 | 설명 |
|------|------|
| **키 (Key)** | 값에 접근하기 위한 고유 식별자. 텍스트 또는 해시 값 형태 |
| **값 (Value)** | 키에 연결된 데이터. 문자열, 리스트, 객체 등 다양한 형태 가능 |
| **키-값 쌍 (Pair)** | 키와 값의 연결 관계 단위 |

### 1.2 키 형태 예시

- 텍스트 키: `last_logged_in_at`
- 해시 키: `253DDEC4`

> 성능 최적화를 위해 키는 짧을수록 유리하다.

---

## 2. 기능적 요구사항 (Functional Requirements)

### 2.1 기본 연산 API

시스템은 다음 두 가지 핵심 연산을 반드시 지원해야 한다.

| 연산 | 시그니처 | 설명 |
|------|----------|------|
| **저장** | `put(key, value)` | 키-값 쌍을 저장소에 저장한다 |
| **조회** | `get(key)` | 주어진 키에 해당하는 값을 반환한다 |

### 2.2 키(Key) 요구사항

- 키는 저장소 내에서 **유일(Unique)** 해야 한다
- 동일한 키로 `put()` 호출 시 기존 값을 **덮어쓴다**
- 값은 반드시 **키를 통해서만** 접근 가능하다
- 키의 형태는 일반 텍스트 또는 해시 값 모두 허용한다

### 2.3 값(Value) 요구사항

- 값의 형태에 제한을 두지 않는다 (문자열, 리스트, 객체 등)
- 키-값 쌍 하나의 크기는 **10KB 이하**여야 한다
- 저장소는 값의 내부 구조를 해석하거나 처리하지 않는다 (Opaque Value)

---

## 3. 비기능적 요구사항 (Non-Functional Requirements)

### 3.1 대용량 데이터 저장

- 단일 서버의 메모리 한계를 초과하는 **대규모 데이터**를 저장할 수 있어야 한다
- 데이터는 여러 서버에 **수평 분산(Horizontal Partitioning)** 되어 저장된다

### 3.2 고가용성 (High Availability)

- 시스템 일부에 **장애가 발생하더라도 서비스가 지속**되어야 한다
- 장애 상황에서도 **빠른 응답**을 보장해야 한다
- 목표 가용성: **99.9% 이상**

### 3.3 높은 규모 확장성 (High Scalability)

- 트래픽 증가에 따라 **서버를 자동으로 증설**할 수 있어야 한다
- 트래픽 감소 시 **서버를 자동으로 제거**할 수 있어야 한다
- 스케일 아웃/인 과정에서 서비스 중단이 없어야 한다

### 3.4 데이터 일관성 조정 가능 (Tunable Consistency)

- 사용 목적에 따라 **일관성 수준을 조정**할 수 있어야 한다
- 지원해야 할 일관성 모델:

| 일관성 수준 | 설명 | 적합한 상황 |
|------------|------|------------|
| **강한 일관성** | 모든 노드에서 항상 최신 값 반환 | 금융, 결제 시스템 |
| **최종 일관성** | 일정 시간 후 모든 노드가 동일한 값으로 수렴 | SNS, 로그 수집 |
| **약한 일관성** | 최신 값 보장 없음, 최고 성능 | 실시간 게임, 캐시 |

### 3.5 짧은 응답 지연시간 (Low Latency)

- **읽기/쓰기 연산 모두** 낮은 응답 지연시간을 유지해야 한다
- 목표 응답 시간: **p99 기준 10ms 이하** (단순 get/put 연산)

---

## 4. 제약 조건 (Constraints)

| 항목 | 제약 내용 |
|------|-----------|
| **키-값 크기** | 하나의 키-값 쌍은 10KB를 초과할 수 없음 |
| **키 유일성** | 동일한 키는 저장소 내에서 하나만 존재 가능 |
| **값 접근 방식** | 값은 반드시 키를 통해서만 접근 가능 (범위 쿼리 등 미지원) |

---

## 5. 트레이드오프 고려사항

완벽한 설계는 존재하지 않는다. 다음 세 축 사이의 균형을 요구사항에 맞게 결정해야 한다.

```
         읽기 성능
            ▲
            │
            │
쓰기 성능 ◄──┼──► 메모리 사용량
            │
            │
         일관성 ↔ 가용성
```

### 5.1 CAP 이론 기반 선택

> 분산 시스템에서 **일관성(C)**, **가용성(A)**, **파티션 허용성(P)** 중 두 가지만 동시에 보장 가능하다.

본 시스템은 **가용성(A) + 파티션 허용성(P)** 을 우선시하되, 일관성 수준은 **사용자가 조정 가능**하도록 설계한다.
