
## 1. 요구사항 정리 및 분석
* **데이터 크기:** 키-값 쌍의 크기는 **10KB 이하**
* **규모 확장성:** 트래픽에 따라 자동으로 서버 증설/삭제가 가능한 분산 구조
* **고가용성:** 일부 서버 장애 시에도 시스템은 멈추지 않고 빠르게 응답해야 함
* **일관성:** 서비스 특성(요구사항)에 맞게 데이터 일관성 수준을 조절 가능해야 함
* **성능:** 응답 지연 시간(Latency)이 짧아야 함

---

## 2. 핵심 API 설계
클라이언트가 실제 저장소와 통신하기 위한 인터페이스.

* `boolean put(String key, String value)`: 데이터를 저장하거나 업데이트. 성공 여부 반환.
* `String get(String key)`: 데이터를 조회. 없으면 `null` 반환.
* `boolean remove(String key)`: 특정 키에 해당하는 데이터를 삭제. 성공 여부 반환.

## 3. 전체 아키텍처 (데이터 분산과 복제)
대규모 트래픽과 데이터를 감당하기 위한 뼈대.

### 3.1. 안정 해시 (Consistent Hashing)
* **도입 이유:** 일반적인 모듈러 해시(`hash(key) % 서버 수`)는 서버 증설 및 장애 시 대규모 데이터 이동(재배치)이 발생하는 치명적인 단점이 있음.
	* MYSQL 샤딩을 개발자가 직접 구현한 경우에는, 개발자가 직접 마이그레이션을 진행해야함.
	* Redis Cluster에 노드를 추가하는 경우에는 Redis가 내부적으로 해시 슬롯 재분배
* **해결책:** 해시 값의 범위를 거대한 둥근 링(Ring)으로 구성(treeMap 자료구조를 사용해서 실제 서버 ip의 해시값을 할당)하고, 서버와 데이터를 동일한 링 위에 배치. 데이터를 저장할 때 데이터 위치에서 **시계 방향으로 걷다가 가장 먼저 만나는 서버**에 할당.
#### 실제로 어떤 과정으로 이루어지는지?
- 서버 IP 해시값을 TreeMap에 저장
- 데이터 요청 → 해시값 계산 → ceilingEntry()로 가장 가까운 서버 탐색
- 실제 서버로 요청 전송
- 가상 노드 = IP#1, IP#2... 해시 → TreeMap에 key로, value는 실제 IP
- 가상 노드 찾으면 value(실제 IP)로 요청 전송



![Consistent Hashing Ring](images/Pasted%20image%2020260606003537.png)

* **효과:** 서버가 추가되거나 삭제될 때, 링 위에서 인접한 위치의 최소한의 데이터만 이동하므로 뛰어난 규모 확장성 확보.
* **요구사항인 규모 확장성, 고가용성, 성능을 만족**
	* 규모확장성 - 서버를 추가하거나 삭제할 때 전체 데이터를 재배치 하지 않고 최소한의 데이터만 이동하니까 서버 증설 삭제가 자유롭다.
	* 고가용성 - 서버 1대가 죽어서 링에서 빠져도, 일부 데이터만 다음 서버로 자동으로 넘어간다.
	* 성능 - 데이터가 어느 서버에 있는지 O(logN)만에 찾을 수 있다.

### 3.2. 데이터 복제 (Replication)
* **도입 이유:** 데이터를 단일 서버에만 저장하면 해당 서버 장애 시 데이터가 영구 유실되고 시스템이 멈춤.
* **해결책:** 안정 해시 링 위에서 시계 방향으로 만나는 **첫 $N$개의 서버에 데이터를 복사**하여 저장. (일반적으로 $N=3$ 사용)

#### ⚠️ 가상 노드 사용 시 복제 주의점 (데이터 편중 문제)
* **문제점:** 링 위에 배치된 가상 노드들을 단순히 시계 방향으로 $N$개 선택할 경우, 동일한 물리 서버에 속한 가상 노드들이 연속으로 선택되어 실제로는 데이터가 1~2대의 물리 서버에만 중복 저장되는 고가용성 위반 문제가 발생할 수 있다.
* **해결책 (고유 물리 서버 필터링):** * 조정자 노드가 링을 순회할 때, 단순히 가상 노드 개수를 세는 것이 아니라 **서로 다른 고유한(Distinct) 물리 서버 $N$개가 확보될 때까지** 이미 선택된 물리 서버의 가상 노드는 건너뛰도록(Skip) 알고리즘을 설계해야 한다.
* **실무적 확장 (랙/데이터 센터 인지):**
  * 더 높은 가용성을 위해 동일 물리 서버뿐만 아니라 동일한 전력/네트워크를 공유하는 물리적 랙(Rack. 데이터 센터에 있는 캐비닛 하나 -> 여러개의 서버 기기가 탑재되어 있음. 전력을 공유하므로, 전력에 문제가 생길 경우, 해당 랙에 있는 서버기기들은 다 영향 받을 수 있음)이나 가용 영역(AZ, Availability Zone. 건물 단위. 서버 기기들, 랙들이 모여있는 건물)까지 고려하여 복제본을 물리적으로 완전히 격리된 공간에 분산 저장함.

## 4. 데이터 일관성 및 충돌 해결
서로 분산된 복제본(Replica)들 사이에서 최신 데이터를 유지하고 충돌을 방지하는 전략.

### 4.1. 정족수 합의 (Quorum Consensus)
공식 `W + R > N`을 사용하여 상황에 맞게 일관성 수준을 조절.
* `N`: 전체 복제본 개수
* `W`: 쓰기(`put`) 연산 시 성공 확답을 받아야 하는 최소 서버 수
* `R`: 읽기(`get`) 연산 시 교차 검증을 위해 데이터를 받아와야 하는 최소 서버 수
* **핵심 원리:** `W + R > N`을 만족하도록 설정하면, 읽기 요청 시 응답을 가져오는 `R`개의 서버 중 최소 1곳은 반드시 최신 쓰기(`W`)가 반영된 서버임(비둘기집의 원리). 
  조정자(Coordinator)가 여러 응답의 버전을 비교해 최신 데이터만 반환하므로 **강한 일관성** 보장.
```
쓰기 성공 = 최소 2개 서버에 저장 완료
읽기 범위 = 최소 2개 서버에서 읽기

2 + 2 = 4 > 3(전체 서버 수)

→ 겹치는 서버가 반드시 최소 1개 존재!
```
#### 예시
- 예를 들어 3개의 서버가 존재한다면, 아래 데이터 형식과 같이 version 을 포함해서 최소 2개의 서버에 저장 완료를 해야 한다.

그 후에 3개의 서버중에서 2개의 서버를 조회했을때 아래와 같이 결과가 나온다면, version을 통해 어떤 데이터가 최신 데이터인지 알 수 있을 것.
```
Server B → { key: "apple", value: "맛있다v2", version: 2 } 
Server C → { key: "apple", value: "맛있다v1", version: 1 } 
version 비교 → 2가 더 크니까 "맛있다v2" 반환
```
* **조절 예시:** 일관성보다 응답 속도가 최우선이라면 `W=1, R=1`로 설정하여 가장 먼저 도착하는 응답을 바로 반환 (최종 일관성).

### 4.2. 벡터 시계 (Vector Clock)를 이용한 충돌 해결

> 위에서 version 값을 통해서 최신여부를 판단한다고 했지만, 만약 서로 다른 클라이언트가 동시에 데이터 수정 요청을 보냈을때는 어떤 데이터가 최신인지 어떻게 구분할까?

* **도입 이유:** 분산 서버 환경에서는 서버 간 물리적 시계 불일치(Clock Drift)가 발생하므로, 단순 시간(Timestamp)으로 최신 여부를 판단하면 데이터 유실 위험이 높음.
* **해결책:** `[서버 이름, 수정 횟수 카운터]` 형태의 논리적 시계(벡터 시계)를 데이터의 꼬리표로 부착하여 버전 관리.
```
{
  "key": "apple",
  "value": "맛있다",
  "vector_clock": {
    "Server A": 2,
    "Server B": 1
  }
}
```

* **충돌 처리:** 만약 조회된 두 데이터의 벡터 시계가 서로 포함 관계가 아니라 엇갈려 있다면(동시에 수정되어 충돌 발생), 시스템이 임의로 판단하지 않고 클라이언트에게 두 데이터를 모두 반환하여 애플리케이션 단에서 병합(Merge)하도록 위임.

벡터 시계는 절대 유실이 되면 안되는 서비스에 적합

### 4.3 LLW(Last write wins)
물리적 서버 시계 기준으로 가장 마지막에 쓰여진 데이터가 최종 데이터가 됨.

- 장바구니와 같은 도메인에는 적합하지 않음. 
	- 고객이 폰으로는 마우스를 담고, PC로는 키보드를 거의 동시에 담았다면, 사실 이 순서와 상관없이 해당 데이터가 해시 링 위에서 배정되는 서버의 물리적 시간에 의해 가장 최종 데이터가 쓰기가 됨.

## 5. 장애 감지 및 대응 (Failure Detection)
중앙 통제 서버 없이 수많은 분산 서버들끼리 서로의 상태를 파악하는 방법.

### 가십 프로토콜 (Gossip Protocol)
* 각 서버가 1초마다 무작위로 주변 서버들을 선택해 자신의 하트비트(Heartbeat) 카운터와 자신이 아는 이웃 서버들의 상태 정보를 소문내듯 전파.
* 특정 서버의 하트비트 카운터가 오랫동안 갱신되지 않는다는 소문이 퍼지고 다수의 서버가 동의하면, 해당 서버를 '장애' 상태로 확정.
* 장애 감지 후, 링 위에서 해당 노드를 제외하고 부족해진 복제본을 인접 서버에 새로 복사하여 가용성을 스스로 복구함.
> 단 가십 프로토콜은 일시적으로 장애가 발생했을때 사용가능한 방법이다. 그럼 영구적인 장애가 발생했을때는 어떻게 해야할까?


### 반-엔트로피 프로토콜(anti-entropy)
- 엔트로피는 물리개념에서 무질서함을 뜻하므로, 무질서함을 없애고 바로잡자는 의미의 프로토콜

#### 버킷이란?
거대한 도서관(서버 데이터 전체)을 통째로 비교하는 대신, 데이터를 구역별로 쪼개는 것.

- **1번 버킷:** 해시값이 1 ~ 1,000인 데이터들만 모아둔 바구니
- **2번 버킷:** 해시값이 1,001 ~ 2,000인 데이터들만 모아둔 바구니
- ... 이런 식으로 수백만 개의 데이터를 예를 들어 256개의 버킷(바구니)에 나누어 담는다.

#### 머클트리란?
버킷으로 쪼개긴 했지만, 여전히 버킷 안의 내용물을 다 꺼내서 비교하는 건 비효율적이다. 
그래서 해시(Hash) 함수의 성질("내용이 1글자라도 다르면 완전히 다른 값이 나온다")을 이용해 머클 트리(Merkle Tree)라는 나무 구조를 만든다.

**머클 트리를 만드는 4단계**
- **데이터 해싱:** 버킷 안에 있는 데이터(키)들을 각각 해시 함수에 넣어서 고유한 해시값을 만듭니다. (나뭇잎)
- **버킷 해싱:** 한 버킷 안에 들어있는 4개의 데이터 해시값을 하나로 합쳐서 또 해시를 돌립니다. 이것이 **'버킷의 해시값'이 된다.
- **위로 올라가며 해싱 (가지):** 인접한 2개의 버킷 해시값을 하나로 합쳐서 또 해시를 만듭니다.
- **루트 해시 (뿌리):** 이 행위를 계속 위로 2개씩 묶어서 올라가다 보면, 최종적으로 꼭대기에는 단 1개의 해시값(Root Hash)만 남게 된다.
