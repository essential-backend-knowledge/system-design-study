## 0. 요구사항에 대한 정량 스펙

| 요구사항 | 정량 기준 (제안) |
|---|---|
| 값 크기 | ≤ 10KB (약 5,000 ~ 10,000자 텍스트 정도 → 작음 → 키가 많은 상황) |
| 전체 규모 | 수십억 ~ 수조 키, 수백 TB ~ PB → 파티션 필수|
| 가용성 | 99.99% (연 ~52분 다운) |
| 확장성 | 수평·선형 확장, 자동 오토스케일 |
| 일관성 | W/R/N 조정 가능 (요청별 레벨 선택) |
| 지연 | p99 < 10ms 목표 |

## 1. 설계 방향


| 요구사항 | 기법 |
|---|---|
| 큰 데이터 저장 / 규모 확장성 | 데이터 파티션 |
| 높은 가용성 / 빠른 복구 | 데이터 다중화 + 장애 처리 |
| 일관성 수준 조정 가능 | 정족수 기반 일관성 |
| 짧은 응답 지연 | 비동기 다중화, 읽기/쓰기 경로 최적화 |

핵심 결정: **AP 시스템**(CAP에서 가용성, 파티션 내성 우선). 강한 일관성 대신 **최종 일관성** 수용. 대신 일관성 수준을 *조정 가능*하게 만들기.

| 항목 | 내용 |
|---|---|
| **선택지** | CP(일관성+분단내성, 예: HBase/Zookeeper) vs **AP(가용성+분단내성, 예: Dynamo/Cassandra)** |
| **고른 것** | **AP** |
| **포기한 것** | 네트워크 분단 시 "항상 최신값" 보장 (강한 일관성) |
| **근거** | 목표가 **가용성 99.99%**, **p99 < 10ms**. CP를 택하면 분단 발생 시 일부 노드가 요청을 거부해야 해서 가용성·지연이 무너짐. 대신 최종 일관성을 받아들이되, 정족수로 일관성을 *끌어올릴 수 있게* 함. |

## 1. 데이터 파티션 (Partition)

**문제**: 데이터가 한 서버에 안 들어가니 여러 노드에 쪼개야 함. 단순 `hash(key) % N` 은 노드 추가/삭제 시 거의 모든 키가 재배치됨.

**해결: 일관성 해싱(Consistent Hashing)**
- 키와 노드를 모두 동일한 해시 링에 매핑.
- 키는 시계 방향으로 만나는 첫 번째 노드에 저장.
- 노드 추가/삭제 시 **인접 구간의 키만** 이동 → 재배치 최소화.

**가상 노드(Virtual Node)**
- 물리 노드 하나를 링 위 여러 지점에 배치.
- 효과
  - 데이터가 고르게 분산(hot spot 완화)
  - 서버 성능 차이를 가상 노드 개수로 반영 가능

→ 요구사항 2(큰 데이터)·4(자동 증설/삭제) 충족.
### 트레이드오프

| 항목 | 내용 |
|---|---|
| **선택지** | `hash(key) % N` vs **일관성 해싱** |
| **고른 것** | **일관성 해싱 + 가상 노드** |
| **포기한 것** | 구현 단순함. (링 관리·가상노드 메타데이터라는 복잡도를 떠안음) |
| **근거** | 모듈로는 노드 추가 시 거의 전체 키 재배치 → **요구사항 4(자동 증설/삭제)** 불가능. 링 방식은 인접 구간만 이동. **PB급·오토스케일** 환경엔 필수. |

**가상 노드 개수 자체도 트레이드오프**:
- 많이(예: 노드당 200~256개) → 부하 분산 고름, 하지만 **메타데이터·가십 부담↑**.
- 적게 → 가볍지만 hot spot·불균형 위험.

## 2. 데이터 다중화 (Replication)

**문제**: 노드 하나 죽으면 데이터 유실 + 가용성 저하.

**해결**: 각 키를 링에서 시계 방향으로 만나는 **첫 N개의 (물리) 노드**에 복제.
- 가상 노드를 쓰면 N개를 고를 때 *같은 물리 노드*가 중복되지 않도록 건너뛰어야 함.
- 더 높은 신뢰성을 위해 N개를 서로 다른 데이터센터/랙에 분산 배치 (지리적 다중화).

→ 요구사항 3(가용성) 충족

## 3. 일관성 (Consistency) - 정족수 기반

- **N** = 복제본 개수
- **W** = 쓰기 정족수 (쓰기 성공으로 인정할 응답 수)
- **R** = 읽기 정족수 (읽기 시 응답을 모을 노드 수)

**핵심 규칙: `W + R > N` 이면 강한 일관성 보장** (읽기 집합과 쓰기 집합이 반드시 겹침 → 최신 값을 최소 1개 포함).

조정 예시

| 설정 | 성격 |
|---|---|
| W=1, R=1 | 빠른 응답, 약한 일관성 |
| W=N, R=1 | 읽기 빠름, 쓰기 느림 |
| R=N, W=1 | 쓰기 빠름, 읽기 느림 |
| W+R>N (예 N=3,W=2,R=2) | 균형, 강한 일관성 |

→ 이 W/R/N을 노출함으로써 요구사항 5(**일관성 수준 조정**) 충족. 요구사항 6(짧은 지연)은 W/R를 낮추거나 가장 빠른 응답만 기다리는 식으로 튜닝.

**일관성 모델**: 기본은 최종 일관성(eventual consistency).

## 4. 일관성 불일치 해소 (Conflict Resolution)

비동기 복제 + 동시 쓰기 → 같은 키에 서로 다른 버전이 생김. "누가 최신인가?"를 풀어야 함.

**벡터 시계(Vector Clock)**
- 값에 `[(서버, 버전카운터)]` 리스트를 붙임. 예: `D([s1,1],[s2,1])`.
- 쓰기마다 해당 서버의 카운터 증가.
- 두 버전 비교 시:
    - 한쪽이 다른 쪽을 **포함**(모든 카운터가 ≥) → 조상-자손 관계, 최신 것 채택.
    - 서로 **포함 못 함** → 충돌(conflict) → 클라이언트가 병합(예: 장바구니 합집합).
- 단점: 리스트가 길어질 수 있음 → 임계치 넘으면 오래된 쌍 제거.
