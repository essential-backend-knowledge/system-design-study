# 키-값 저장소 (Key-Value Store) 시스템 아키텍처 문서

> **문서 목적**: 요구사항 정의서를 기반으로 분산 키-값 저장소의 전체 시스템 구조와 각 컴포넌트의 역할 및 상호작용을 정의한다.  
> **참조 문서**: `key-value-store-requirements.md`  
> **설계 원칙**: 가용성(A) + 파티션 허용성(P) 우선, 일관성은 조정 가능

---

## 1. 아키텍처 개요

### 1.1 설계 목표 요약

| 요구사항 | 아키텍처 결정 |
|---------|-------------|
| 대용량 데이터 저장 | 데이터를 여러 노드에 수평 분산 (파티셔닝) |
| 고가용성 (99.9%) | 모든 데이터를 N개 노드에 복제 |
| 낮은 응답 지연 | 인메모리 버퍼 + 블룸 필터로 읽기 최적화 |
| 규모 확장성 | 안정 해시(Consistent Hashing)로 서버 추가/제거 시 재배치 최소화 |
| 일관성 조정 | W / R / N 쿼럼 값으로 일관성 수준 제어 |
| 장애 감지 | 가십 프로토콜(Gossip Protocol)로 자율적 상태 공유 |

### 1.2 전체 아키텍처 구성도

## 2. 계층별 컴포넌트 상세

### 2.1 클라이언트 계층

클라이언트는 `put(key, value)` 및 `get(key)` API를 통해 시스템과 통신한다. 클라이언트는 내부 구현에 대한 지식 없이 코디네이터 노드에 요청을 보내기만 하면 된다.

**클라이언트가 알아야 하는 것:**
- 코디네이터 노드의 엔드포인트 주소
- 요청 시 원하는 일관성 수준 (선택적)

**클라이언트가 알 필요 없는 것:**
- 데이터가 실제로 저장된 노드
- 복제 구조
- 파티셔닝 로직

---

### 2.2 코디네이터 계층

클라이언트 요청을 받아 적절한 스토리지 노드로 라우팅하는 중간 계층이다. 코디네이터 자체는 상태를 저장하지 않으며(Stateless), 어떤 노드든 코디네이터 역할을 수행할 수 있다.

#### 코디네이터의 주요 역할

| 역할 | 설명 |
|------|------|
| **요청 수신** | 클라이언트로부터 put / get 요청 수신 |
| **노드 결정** | 안정 해시를 통해 담당 스토리지 노드 선택 |
| **쿼럼 조율** | W / R 값에 따라 필요한 노드 수만큼 요청 분배 및 응답 수집 |
| **장애 감지** | 멤버십 정보를 통해 현재 활성 노드 목록 파악 |
| **응답 반환** | 수집된 응답을 클라이언트에게 전달 |

#### 쿼럼(Quorum) 동작 방식

쿼럼은 세 가지 값으로 일관성 수준을 조절한다.

```
N = 복제본 총 수
W = 쓰기 성공으로 인정하기 위해 필요한 최소 응답 수
R = 읽기 성공으로 인정하기 위해 필요한 최소 응답 수
```

| 설정 | 특성 | 적합한 용도 |
|------|------|------------|
| W=1, R=N | 빠른 쓰기, 느린 읽기 | 로그 수집, 이벤트 기록 |
| W=N, R=1 | 느린 쓰기, 빠른 읽기 | 캐시, 조회 중심 서비스 |
| W+R > N | 강한 일관성 보장 | 금융, 결제 |
| W+R ≤ N | 최종 일관성, 최고 성능 | SNS, 게임 |

---

### 2.3 스토리지 노드 계층

실제 데이터를 저장하고 반환하는 계층이다. 모든 노드는 동일한 구조를 가지며(Peer-to-Peer), 중앙 단일 장애점(SPOF)이 없다.

#### 스토리지 노드 내부 구조

```
쓰기 요청 (put)
      │
      ▼
┌─────────────┐    WAL(Write-Ahead Log)에 먼저 기록
│  WAL 기록   │  → 장애 발생 시 복구 보장
└──────┬──────┘
       │
       ▼
┌─────────────┐    메모리에 키-값 저장
│ 메모리 버퍼  │  → 빠른 쓰기 성능 확보
│  (Memtable) │  → 일정 크기 초과 시 디스크로 플러시
└──────┬──────┘
       │ 임계치 초과
       ▼
┌─────────────┐    정렬된 형태로 디스크에 저장
│  SSTable    │  → 불변(Immutable) 파일
│   (Disk)    │  → 주기적으로 컴팩션(Compaction) 수행
└─────────────┘

읽기 요청 (get)
      │
      ▼
┌─────────────┐    키 존재 여부를 확률적으로 빠르게 확인
│  블룸 필터  │  → 없으면 디스크 탐색 불필요 (불필요한 I/O 방지)
└──────┬──────┘
       │ 존재 가능성 있음
       ▼
┌─────────────┐
│ 메모리 버퍼  │ → 있으면 즉시 반환
└──────┬──────┘
       │ 없음
       ▼
┌─────────────┐
│  SSTable    │ → 최신 SSTable부터 순서대로 탐색
└─────────────┘
```

#### 각 내부 컴포넌트 역할 요약

| 컴포넌트 | 역할 | 특성 |
|---------|------|------|
| **WAL** | 쓰기 내구성 보장 | 순차 기록, 빠른 쓰기 |
| **Memtable** | 인메모리 쓰기 버퍼 | 정렬된 트리 구조 (예: Red-Black Tree) |
| **SSTable** | 디스크 영구 저장소 | 불변, 정렬 상태 유지 |
| **블룸 필터** | 키 존재 여부 빠른 판별 | 확률적 자료구조, False Positive 허용 |
| **Compaction** | SSTable 병합 및 정리 | 삭제된 키 제거, 읽기 성능 향상 |

## 3. 분산 메커니즘

### 3.1 데이터 파티셔닝 — 안정 해시 (Consistent Hashing)

키를 여러 노드에 균등하게 분산하기 위해 안정 해시를 사용한다.

```
해시 링 (Hash Ring)
             0
           ┌───┐
    Node A  │   │  Node B
   (0~90)  ←┤   ├→ (91~180)
            │   │
    Node D  │   │  Node C
  (271~360) └───┘ (181~270)
            360/0

키 "user:1001" → 해시값 150 → Node C 담당
키 "user:2345" → 해시값 45  → Node A 담당
```

**안정 해시의 장점:**
- 노드 추가/제거 시 전체 키 중 일부만 재배치 (기존 방식 대비 최소화)
- 각 노드가 가상 노드(Virtual Node)를 여러 개 가져 부하 균등 분산

---

### 3.2 데이터 복제 (Replication)

높은 가용성을 위해 동일한 데이터를 N개의 노드에 복제 저장한다.

```
put("user:1001", "Alice")
         │
    [코디네이터]
    (해시 → Node C 담당)
         │
    ┌────┴──────────┐
    │               │
[Node C]        [Node D]        [Node A]
(Primary)    (Replica 1)    (Replica 2)
   저장           저장            저장
    │               │               │
  성공             성공            성공
    └────────────────┴───────────────┘
              W=2 이상 성공 시
           클라이언트에 완료 응답
```

- 복제는 해시 링에서 해당 노드의 **시계 방향으로 N개 노드**에 수행
- W / R 값 조절로 일관성과 성능 균형 조정

---

### 3.3 장애 감지 — 가십 프로토콜 (Gossip Protocol)

중앙 집중식 장애 감지 대신, 노드들이 서로 상태 정보를 주기적으로 교환한다.

```
매 초마다 각 노드는 랜덤으로 다른 노드 k개를 선택해 자신의 멤버십 정보를 전달

Node A → Node C: "나는 살아있고, B는 v12, C는 v15 상태야"
Node C → Node D: "나는 살아있고, A는 v8, B는 v12, D는 v20 상태야"
Node B → Node A: "B는 오랫동안 heartbeat가 없어 → 장애로 판단"
```

**장애 판단 기준:**
- 각 노드는 멤버십 테이블에서 상태 버전 번호(heartbeat)를 관리
- 일정 시간 동안 버전이 갱신되지 않으면 해당 노드를 장애 상태로 표시

---

### 3.4 장애 처리 — 임시 쓰기 (Hinted Handoff)

장애 노드에 써야 할 데이터를 다른 노드가 임시 보관하다가, 장애 복구 후 전달한다.

```
Node C가 장애 상태일 때 put("user:1001") 요청

코디네이터 → Node D에 "이건 원래 Node C 것인데 임시로 맡아줘" 라고 힌트와 함께 저장
                       (Hinted Replica)

Node C 복구 후 → Node D가 보관 중이던 데이터를 Node C에 전달 → 삭제
```

## 4. 데이터 흐름

### 4.1 쓰기 흐름 (put)

```
① 클라이언트가 put(key, value) 요청
② 코디네이터가 안정 해시로 담당 노드(Primary) 결정
③ 코디네이터가 Primary + Replica 노드들에 동시 쓰기 요청
④ 각 노드: WAL 기록 → Memtable 저장
⑤ W개 노드로부터 성공 응답 수신
⑥ 클라이언트에 성공 응답 반환
```

### 4.2 읽기 흐름 (get)

```
① 클라이언트가 get(key) 요청
② 코디네이터가 안정 해시로 담당 노드 결정
③ 코디네이터가 R개 노드에 동시 읽기 요청
④ 각 노드:
   - 블룸 필터로 키 존재 여부 확인
   - Memtable 탐색 → 없으면 SSTable 탐색
⑤ R개 응답 수집 후 가장 최신 버전 선택 (버전 벡터 비교)
⑥ 클라이언트에 값 반환
⑦ (선택) Read Repair: 오래된 복제본에 최신 값 동기화
```

## 5. 컴포넌트 책임 요약

| 컴포넌트 | 담당 요구사항 | 핵심 기술 |
|---------|------------|---------|
| 코디네이터 | 요청 라우팅, 일관성 조율 | 안정 해시, 쿼럼 |
| 스토리지 노드 | 데이터 저장 및 반환 | Memtable, SSTable, WAL |
| 블룸 필터 | 읽기 지연시간 최소화 | 확률적 자료구조 |
| 가십 프로토콜 | 장애 감지 및 멤버십 관리 | 분산 상태 전파 |
| 안정 해시 | 데이터 파티셔닝 | 해시 링, 가상 노드 |
| 복제 | 고가용성 보장 | N개 복제본, Read Repair |
| Hinted Handoff | 장애 노드 데이터 보전 | 임시 저장 및 복구 후 전달 |
| Compaction | 디스크 효율화 | SSTable 병합 |

## 6. 설계 결정 및 트레이드오프

### 6.1 중앙 집중식 vs 분산 피어-투-피어

본 시스템은 **피어-투-피어(P2P)** 구조를 채택한다.

| 항목 | 중앙 집중식 | 피어-투-피어 (채택) |
|------|-----------|-----------------|
| 단순성 | 높음 | 낮음 |
| 단일 장애점 | 존재 | 없음 |
| 확장성 | 낮음 | 높음 |
| 가용성 | 낮음 | 높음 |

### 6.2 메모리 vs 디스크 우선

- **쓰기**: 메모리 우선 (Memtable) → 속도 확보
- **내구성**: WAL로 보완 → 메모리 유실 시 복구 가능
- **읽기**: 메모리 → 디스크 순서로 탐색, 블룸 필터로 불필요한 디스크 I/O 차단

### 6.3 일관성 vs 가용성

CAP 이론에 따라 네트워크 파티션 발생 시 **가용성을 우선** 선택한다. 단, W+R > N 조건으로 강한 일관성이 필요한 경우에도 대응 가능하도록 쿼럼 설정을 노출한다.
