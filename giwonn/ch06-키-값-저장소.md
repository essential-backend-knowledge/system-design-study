
### 1. 요구사항 정리
- Key는 유일성을 가지며, 일반 텍스트거나 해시 값일 수도 있다.
- Value는 무엇이 오든 상관하지 않는다.
- `put(key, value)`, `get(key)` 인터페이스를 지원한다.

- 장애가 있어도 빠르게 응답해야 한다 -> 고가용성
- 트래픽에 따라 서버가 자동적으로 증설/삭제가 이루어져야 한다.
- 데이터 일관성 수준은 조정이 가능해야 한다.-> 정확히 어떻게..?
- 응답 지연시간이 짧아야 한다.


### 2. 설계 결정
- put 시 완전히 동일한 문자열 key면 **overriding**한다.

- **key의 해시값이 동일한 경우 체이닝으로 처리**한다.

- **체이닝은 LinkedList 자료구조를 사용**하며, createdAt등의 타임스탬프로 최신 데이터를 결정한다. -> *만약 동일한 시간이 동시에 인입된다면*? -> *쓰기를 싱글스레드로 설계하면 동일한 시간이 존재할 수가 있나?*

- get에서 한 버킷에 여러 노드(체인)가 있으면 **문자열 key 비교**로 값을 특정한다
  (해시값은 버킷 위치만 정하고, 최종 식별은 full key 비교로 한다.)

- 만료된 key는 **lazy** 방식으로 정리한다. (주기적으로 gc하도록)

- put 완료 시 쓰기 명령을 디스크에 **AOF(append-only)** 로 기록해 장애 시 복구 가능하게 한다.

- TTL을 쓴다는 건 "이 값은 최대 TTL초만큼 낡아도 된다"는 뜻 → 분산 시 **AP를 우선** 고려한다.



###  2. 기능 흐름
- **put (write-through)**: 캐시에 적용 + 원본 DB에 동시 기록 → AOF append

- **get**: `hash(key) → 버킷 인덱스 직행 → 체이닝 탐색 → key 비교 → 값 반환`. (조회 시 만료됐으면 null 리턴)
```
key -> k1 (v1), k1 (v2)
```

- **만료 정리(active)**: 타이머로 주기 실행하되, 전체가 아니라 **TTL이 걸린 키만 모은 expires 인덱스**를 시간 내에서 수집해 정리한다. (gc가 너무 오래 점유하지 않도록)

- **저장 구조**: 버킷 충돌 시 **LinkedList 체이닝**으로 충돌 해소

### 4. CAP 트레이드오프
복제를 하려면 네트워크 분할이 현실이라 **P는 버릴 수 없다.**

##### 가용성(A)을 희생한다면? (CP)
분할 시 일관성을 지키려 응답을 포기한다.
최신 데이터를 제공하는 장점이 있으나, **이 캐시엔 부적합**하다.
캐시가 멈추면 트래픽이 DB로 쏟아져 DB 장애 발생할 수 있다.

##### 일관성(C)을 희생한다면? (AP)
분할 시 각 노드가 자기가 가진 값으로 계속 응답한다.
원본이 source of truth라 잠깐 예전 데이터를 서빙해도 정확성은 TTL을 얼마로 잡냐에 따라 금방 보장된다.

##### 만약... 파티션 감내를 희생한다면? (CA)
네트워크 분할이 없다고 가정 = 사실상 **단일 노드**. 진짜 분산 환경에선 분할이 불가피하므로, 복제로 확장하는 순간 CA는 선택지에서 빠진다.


> **결론**: 분산 시 **AP**를 택한다. 쓰기는 단일 리더로 직렬화한다.



### 4. 전체 아키텍처
```
[Client] ── write ───┐                         ┌── read ──┐
                     ▼                         ▼          ▼
            ┌────────────── Leader ──────────────┐  (read 분산)
            │ 1. HashTable 적용 (override)        │  ┌── Follower ──┐
            │ 2. write-through → 원본 DB          │  │ AOF replay   │
            │ 3. AOF append (+ fsync 정책)        │  │ HashTable    │
            └──────────────┬─────────────────────┘  │ 만료 스윕     │
                           │ AOF 스트리밍 = 복제 채널  └──────▲───────┘
                           └────────────────────────────────┘
```



### 5. 병목 또는 장애 가능 지점
- **AOF 쓰기 장애 / 부분 쓰기** - 디스크 기록 중 장애시 마지막 레코드가 잘릴 수 있다. 복구 시 마지막 유효 레코드까지만 읽고 깨진 부분은 제거해야 한다. 미처리 시 잘못된 레코드로 인해 복구가 통째로 실패한다.

- **AOF 로그 무한 증가** - 압축이 없거나 디스크가 차고 재시작 replay 시간이 로그 크기에 비례해 늘어난다. → 쓰기 트래픽이 swap보다 많을 경우 장애가 발생할 수 있음
- -> Kafka, ELK -> 2 week DELETE -> TTL

- **해시 분포 편향** - 특정 패턴의 데이터가 write되면 특정 해시에만 데이터가 몰릴 수 있다
- -> hash(key) % 서버수 -> `hash(key + 랜덤 seed) % 서버수`

- **리더가 죽는 경우** - leader를 하나로 두고 다른 복제본을 leader로 승격 시키는것은 아직 고려하지 않음... -> 합의 알고리즘을 생각해보자.

- **쓰기가 너무 많다면?** - read만 수평확장하는것으로 고려했기 때문에, 쓰기가 너무 많다면 샤딩도 고려 필요



### 6. 이 구조를 선택한 이유
- **해시테이블(배열 버킷) + LinkedList 체이닝**
	- put/get은 가장 자주 호출되는 경로 → 배열 인덱스 직행으로 평균 **O(1)** 조회/삽입.
	- 체인 맨 앞 삽입·연결 해제가 O(1)이라 순회 비용이 사실상 무시된다. 배열 기반 리스트는 중간 삭제 시 뒤 원소를 전부 밀어내는 O(n) 비용이 들어 만료 삭제에 불리하므로 배제했다.
	- 체이닝을 택한 이유: LinkedList 체이닝은 삭제가 연결 해제 한번 하는게 전부라 빠르다. 

- **AOF 방식으로 영속화를 설계한 이유**
    - 순차 append라 랜덤 쓰기보다 쓰기가 빠르다.
    - 구현이 단순하다: put 명령을 그대로 append하고, 재시작 시 같은 처리 코드로 replay하면 상태가 복원된다
    - 스냅샷 대비 손실 구간이 작고, 멈추지 않고 일관된 스냅샷을 떠야 하는 문제를 피한다

- **AP를 택한 이유**
    - TTL을 거는 순간 어느정도 일관성 희생을 이미 허용한 것
	    - 분할 중 멈추면 thundering herd로 원본이 죽으므로 일관성이 깨지더라도 예전 데이터로 원본을 보호한다.
