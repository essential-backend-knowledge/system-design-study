# 처리율 제한 장치 (Rate Limiter) 요구사항 정의서

> **문서 목적**: 분산 환경에서 동작하는 서버 측 처리율 제한 장치 설계를 위한 기능적/비기능적 요구사항 정리
> **작성 기준**: 시스템 규모 — 대규모 트래픽을 처리하는 분산 환경 가정

---

## 1. 개요

처리율 제한 장치(Rate Limiter)는 클라이언트 또는 서비스가 보내는 트래픽의 **처리율(rate)을 제어**하기 위한 장치이다.
HTTP 컨텍스트에서는 특정 기간 내 클라이언트가 전송할 수 있는 요청 횟수를 제한하며, 정의된 임계치(threshold)를 초과한 추가 요청은 처리가 중단(block)된다.

### 1.1 도입 효과

| 효과            | 설명                                               |
| ------------- | ------------------------------------------------ |
| **DoS 공격 방어** | 자원 고갈(resource starvation)을 유발하는 악의적 트래픽을 차단한다   |
| **비용 절감**     | 우선순위가 높은 API에 자원을 할당하고, 과금 기반 외부 API 호출 횟수를 통제한다 |
| **서버 과부하 방지** | 봇 트래픽이나 잘못된 이용 패턴으로 인한 폭주를 사전에 걸러낸다              |

### 1.2 사용 사례 예시

- 사용자는 **초당 2회 이상** 새 글을 올릴 수 없다.
- 같은 IP 주소로는 **하루에 10개 이상**의 계정을 생성할 수 없다.
- 같은 디바이스로는 **주당 5회 이상** 리워드를 요청할 수 없다.

---

## 2. 설계 범위 (Scope)

면접 상황에서의 질의응답을 통해 확정된 설계 범위는 다음과 같다.

| 항목 | 결정 사항 |
|------|-----------|
| **제한 장치 위치** | 서버 측(Server-side) API를 위한 장치 |
| **제어 규칙 유연성** | 다양한 형태의 throttling rule을 정의할 수 있는 **유연한 시스템** |
| **시스템 규모** | 대규모 요청을 처리할 수 있어야 함 |
| **동작 환경** | **분산 환경(Distributed Environment)** 에서 동작 |
| **독립 서비스 여부** | 설계자의 판단에 따라 결정 (독립 서비스 / 애플리케이션 내장 모두 가능) |
| **사용자 피드백** | 제한된 요청에 대해 **사용자에게 명확히 알림** |

---

## 3. 기능적 요구사항 (Functional Requirements)

### 3.1 처리율 제한

- 설정된 처리율(rate)을 **초과하는 요청은 정확하게 제한**해야 한다.
- 제어 규칙(throttling rule)을 **다양한 기준**으로 정의할 수 있어야 한다.
  - 예: IP 주소, 사용자 ID, API Key, 디바이스 식별자 등

### 3.2 예외 처리 및 사용자 피드백

- 요청이 제한되었을 때 사용자에게 **그 사실을 분명하게 보여주어야 한다**.
- HTTP 응답 코드 및 헤더를 통해 제한 상태와 재시도 정보를 전달해야 한다.
  - 표준 응답: `429 Too Many Requests`
  - 부가 정보: `Retry-After`, `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` 등

---

## 4. 비기능적 요구사항 (Non-Functional Requirements)

### 4.1 낮은 응답 시간 (Low Latency)

- 처리율 제한 장치는 **HTTP 응답시간에 나쁜 영향을 주어서는 안 된다**.
- 제한 여부 판단 로직은 요청 처리 경로상의 **최소 오버헤드**로 동작해야 한다.

### 4.2 메모리 효율성

- 가능한 한 **적은 메모리**를 사용해야 한다.
- 분산 카운터 저장소(Redis 등)의 메모리 사용량을 의식적으로 설계해야 한다.

### 4.3 분산형 처리율 제한 (Distributed Rate Limiting)

- 하나의 처리율 제한 장치를 **여러 서버나 프로세스에서 공유**할 수 있어야 한다.
- 분산 환경에서 카운터의 **정합성(consistency)** 이 보장되어야 한다.

### 4.4 높은 결함 감내성 (Fault Tolerance)

- **제한 장치에 장애가 생기더라도 전체 시스템에 영향을 주어서는 안 된다**.
- 장애 상황에서의 fallback 전략(fail-open / fail-close)을 명확히 정의해야 한다.

## 5. 요구사항 요약 표

| 분류 | 요구사항 | 핵심 키워드 |
|------|----------|-------------|
| **기능** | 임계치 초과 요청 정확히 제한 | 정확성(Accuracy) |
| **기능** | 다양한 제어 규칙 정의 가능 | 유연성(Flexibility) |
| **기능** | 제한 시 사용자에게 명확한 알림 | UX, HTTP 429 |
| **비기능** | HTTP 응답시간에 영향 최소화 | Low Latency |
| **비기능** | 적은 메모리 사용 | Memory Efficiency |
| **비기능** | 분산 환경에서 상태 공유 | Distributed |
| **비기능** | 장애가 전체 시스템에 전파되지 않음 | Fault Tolerance |

## 6. 트레이드오프 고려사항

처리율 제한 장치 설계 시 균형을 잡아야 할 주요 축은 다음과 같다.

```
        정확성(Accuracy)
              ▲
              │
              │
  응답속도 ◄──┼──► 메모리 사용량
   (Latency)  │     (Memory)
              │
              ▼
        가용성 ↔ 결함 감내성
```

### 6.1 핵심 트레이드오프

| 선택 축 | A | B | 고려 사항 |
|--------|---|---|----------|
| **제한 정확성** | 엄격한 정확성 (분산 락) | 근사값 허용 (성능 우선) | 카운팅 알고리즘 선택과 직결 |
| **장애 시 정책** | **Fail-Open** (통과 허용) | **Fail-Close** (모두 차단) | 가용성 vs 보호 강도 |
| **카운터 저장소** | 인메모리(고속) | 영속성(내구성) | 응답 속도 vs 데이터 안정성 |
| **제한 단위** | 클라이언트별(세밀) | 전역(단순) | 정밀도 vs 운영 단순성 |

## 8. 다음 설계 단계로 넘어가기 위한 핵심 질문

본 요구사항으로부터 도출되는, 이후 설계 단계에서 결정해야 할 주요 의사결정 항목이다.

1. **처리율 제한 알고리즘**은 무엇을 선택할 것인가?
   - 토큰 버킷 / 누출 버킷 / 고정 윈도 카운터 / 이동 윈도 로그 / 이동 윈도 카운터
2. **처리율 제한 장치의 배치 위치**는 어디로 할 것인가?
   - 클라이언트 / 서버(미들웨어) / API Gateway / 독립 서비스
3. **분산 카운터 저장소**는 무엇을 사용할 것인가?
   - Redis(INCR/EXPIRE) / Memcached / 자체 구현
4. **카운터 키 설계**와 **TTL** 정책은 어떻게 할 것인가?
5. **장애 시 fallback 전략**은 fail-open인가 fail-close인가?
