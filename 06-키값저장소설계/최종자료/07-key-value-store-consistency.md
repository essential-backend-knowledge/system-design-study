# 키-값 저장소 (Key-Value Store) 일관성 모델 설계서

> **문서 목적**: 분산 환경에서 복제본 간 데이터 일관성을 어떻게 정의하고, 수준별로 어떻게 구현하며, 충돌 발생 시 어떻게 해결하는지를 상세히 정의한다.  
> **참조 문서**: `key-value-store-architecture.md`, `key-value-store-data-model.md`, `key-value-store-replication.md`  
> **핵심 결정**: 기본값은 최종 일관성(Eventual Consistency), 쿼럼 설정으로 강한 일관성까지 조정 가능

---

## 1. 일관성 문제가 발생하는 이유

단일 서버에서는 일관성 문제가 없다. 값을 쓰면 즉시 읽을 수 있다. 그러나 복제 환경에서는 복제본이 여러 노드에 퍼져 있어, **쓰기와 읽기 사이에 시간 간격**이 생긴다.

```
시나리오: N=3 환경에서 put("balance", "1000") 직후 get("balance")

  t=0ms   클라이언트 A: put("balance", "1000")
  t=2ms   Node B 저장 완료 (Primary)
  t=3ms   Node C 저장 완료 (Replica 1) ← W=2 충족, 클라이언트 A에 성공 응답
  t=5ms   클라이언트 B: get("balance") → Node D(Replica 2)에서 읽기
  t=8ms   Node D 저장 완료 (Replica 2)

  문제: t=5ms 시점에 Node D는 아직 "1000"을 받지 못했다
        클라이언트 B가 이전 값("0")을 읽을 수 있다 ← 일관성 문제
```

이처럼 분산 시스템에서 일관성은 **"얼마나 최신 데이터를 보장하느냐"** 의 문제다.

## 2. CAP 이론과 본 시스템의 선택

### 2.1 CAP 이론 요약

분산 시스템은 네트워크 파티션(P) 상황에서 일관성(C)과 가용성(A) 중 하나를 포기해야 한다.

```
          일관성 (C)
         Consistency
              ▲
              │
              │  CP           CA
              │  (일관성 우선) (단일 서버)
              │
              └─────────────────▶ 가용성 (A)
             /                   Availability
            /
           /  AP
          /   (가용성 우선)
         ▼
   파티션 허용성 (P)
   Partition Tolerance
```

| 선택 | 특성 | 대표 시스템 |
|------|------|-----------|
| **CP** | 파티션 시 일부 요청 거부, 데이터 정합성 보장 | HBase, Zookeeper, etcd |
| **AP** | 파티션 시에도 응답, 일시적 불일치 허용 | Cassandra, DynamoDB, Riak |
| **CA** | 파티션 허용성 포기 (단일 서버만 가능) | 전통적 RDBMS |

### 2.2 본 시스템의 선택: AP 기반, 일관성 수준 조정 가능

```
기본 설계 방향:
  → 가용성(A) + 파티션 허용성(P) 우선 (AP)
  → 단, 쿼럼(W/R/N) 조정으로 CP에 가까운 동작도 가능

이유:
  - 요구사항: "장애가 있더라도 빨리 응답해야 한다" → 가용성 우선
  - 요구사항: "데이터 일관성 수준은 조정이 가능해야 한다" → 유연성 확보
  - 모든 서비스가 강한 일관성을 필요로 하지 않음
```

## 3. 일관성 수준 정의

본 시스템은 세 가지 일관성 수준을 지원한다. 클라이언트는 요청마다 `X-Consistency-Level` 헤더로 수준을 선택할 수 있다.

### 3.1 강한 일관성 (Strong Consistency)

```
설정: W=3, R=3 (N=3 기준, 모든 노드 참여)

동작:
  - 쓰기: N개 노드 전부 저장 완료 후 성공 응답
  - 읽기: N개 노드 전부 조회 후 최신 값 반환
  - 항상 가장 최신 값을 읽는 것이 보장됨

보장 조건:
  W + R = 6 > N = 3 → 쓰기 집합과 읽기 집합이 반드시 전부 겹침
  → 어떤 노드에서 읽어도 최신 값

시나리오:
  t=0ms  put("balance", "1000") → N=3 전부 저장 대기
  t=8ms  Node B, C, D 모두 저장 완료 → 클라이언트에 성공 응답
  t=9ms  get("balance") → N=3 전부 읽기 → "1000" 반환 (항상 최신)

트레이드오프:
  ✅ 항상 최신 값 보장
  ✅ 충돌 발생 불가 (모든 노드 동기화 후 응답)
  ❌ 응답 시간이 가장 느린 노드에 종속됨
  ❌ N개 중 1개라도 장애 시 쓰기 불가 (가용성 저하)

적합한 용도: 금융 잔액, 결제 트랜잭션, 재고 수량, 분산 락
```

### 3.2 최종 일관성 (Eventual Consistency)

```
설정: W=2, R=2 (N=3 기준, 과반수 참여) ← 기본값

동작:
  - 쓰기: W개 노드 저장 완료 후 성공 응답
  - 읽기: R개 노드 조회 후 최신 버전 선택
  - 일시적으로 오래된 값이 반환될 수 있으나
    결국 모든 복제본이 동일한 값으로 수렴함

보장 조건:
  W + R = 4 > N = 3 → 쓰기·읽기 집합이 최소 1개 겹침
  → 항상 최신 값을 포함한 노드가 읽기에 참여
  → 강한 일관성과 거의 동일하나, N개 중 1개 노드 지연 허용

시나리오:
  t=0ms   put("profile:1001", "Alice")
  t=3ms   Node B 저장 완료 (1/2)
  t=5ms   Node C 저장 완료 (2/2) ← W=2 충족, 성공 응답
  t=6ms   get("profile:1001") → Node B, C 조회 → "Alice" 반환 ✅
  t=8ms   Node D 저장 완료 (비동기)

트레이드오프:
  ✅ 강한 일관성과 유사한 보장 (W+R>N 조건 유지)
  ✅ 노드 1개 장애/지연에도 서비스 유지
  ⚠️ 이론적으로 극히 드문 경우 오래된 값 반환 가능
  ✅ 강한 일관성보다 빠른 응답

적합한 용도: 사용자 프로필, 상품 정보, 일반 설정값
```

### 3.3 약한 일관성 (Weak Consistency)

```
설정: W=1, R=1 (N=3 기준, 최소 참여)

동작:
  - 쓰기: 1개 노드 저장 완료 즉시 성공 응답
  - 읽기: 1개 노드에서 즉시 반환 (최신 값 보장 없음)
  - 오래된 값이 반환될 수 있으며, 이를 허용함

보장 조건:
  W + R = 2 ≤ N = 3 → 쓰기·읽기 집합이 겹치지 않을 수 있음
  → 최신 값이 없는 노드가 읽기에 참여할 수 있음

시나리오:
  t=0ms   put("session:abc", "user:1001")
  t=1ms   Node B 저장 완료 (1/1) ← W=1 충족, 즉시 성공 응답
  t=2ms   get("session:abc") → Node D 조회 (아직 복제 안 됨)
           → 이전 값 또는 null 반환 ⚠️
  t=3ms   Node C, D 저장 완료 (비동기)
  t=4ms   get("session:abc") → Node D → "user:1001" 반환 ✅

트레이드오프:
  ✅ 가장 빠른 응답 (1개 노드만 참여)
  ✅ 노드 2개 장애에도 서비스 유지
  ❌ 오래된 값 반환 가능 (허용 범위 내)
  ❌ W+R ≤ N → 데이터 손실 가능성 존재

적합한 용도: 세션 캐시, 임시 토큰, 실시간 게임 상태, 로그 수집
```

### 3.4 세 수준 비교 요약

```
                  약한       최종        강한
                  일관성     일관성      일관성
                   W=1        W=2         W=3
                   R=1        R=2         R=3
                ────────────────────────────────
응답 속도        ████████   █████       ███
쓰기 가용성      ████████   █████       ███
읽기 최신성      ███        █████       ████████
충돌 가능성      높음        낮음        없음
노드 장애 허용   N-1개       N/2개       0개
```

## 4. 버전 관리와 충돌 감지

### 4.1 단조 증가 버전 번호

모든 쓰기 연산은 버전 번호를 1씩 증가시킨다. 버전 번호는 복제본 간 최신 값을 판별하는 기준이다.

```
초기 상태:   put("user:1001", "Alice")  → version: 1
첫 번째 수정: put("user:1001", "Alice2") → version: 2
두 번째 수정: put("user:1001", "Alice3") → version: 3

버전 비교:
  Node B: user:1001 = "Alice3", version=3  ← 최신
  Node C: user:1001 = "Alice2", version=2  ← 오래된 복제본
  Node D: user:1001 = "Alice3", version=3  ← 최신

  version이 높은 쪽이 최신 → "Alice3" 채택
```

### 4.2 충돌이 발생하는 조건

단순 버전 번호만으로는 **동시 쓰기(Concurrent Write)** 를 판별하기 어렵다.

```
시나리오: 네트워크 파티션 중 두 클라이언트가 동시에 쓰기

  파티션 발생: [Node B, C] | [Node D] 분리

  클라이언트 X → Node B, C: put("counter", "10")  version: 5
  클라이언트 Y → Node D:    put("counter", "20")  version: 5

  파티션 복구 후:
    Node B: "10", version=5
    Node C: "10", version=5
    Node D: "20", version=5

  ← 두 값의 version이 같음 → 어느 것이 최신인지 알 수 없음 → 충돌
```

## 5. 충돌 해결 전략

### 5.1 Last-Write-Wins (LWW) — 기본 정책

타임스탬프가 더 늦은 쓰기를 최신으로 채택한다.

```
동작:
  Node B: "10", version=5, timestamp=1000ms
  Node D: "20", version=5, timestamp=1003ms  ← 더 늦은 타임스탬프

  → "20" 채택, "10"은 폐기

장점:
  - 구현이 단순하고 추가 메타데이터가 적음
  - 성능 오버헤드 없음

단점:
  - 클럭 동기화 의존 (NTP 오차 존재)
  - 타임스탬프가 같을 경우 임의 선택 필요
  - 데이터 유실 발생 가능 (클라이언트 X의 쓰기 폐기됨)

적합한 상황:
  - 캐시, 세션, 로그 등 유실이 허용되는 데이터
  - 약한 일관성(weak) 수준 선택 시
```

### 5.2 벡터 클록 (Vector Clock) — 충돌 추적 정책

노드별 쓰기 횟수를 배열로 추적하여 인과 관계를 파악한다.

```
벡터 클록 형식: { 노드ID: 쓰기 횟수 }

초기 상태:
  클라이언트 → Node A: put("val", "Alice")
  Node A가 B, C로 복제

  Node A: "Alice", vc={A:1}
  Node B: "Alice", vc={A:1}  ← A가 쓴 것 복제됨
  Node C: "Alice", vc={A:1}

순차 수정 (인과 관계 명확):
  클라이언트 → Node A: put("val", "Alice2")
  Node A: "Alice2", vc={A:2}
  Node B: "Alice2", vc={A:2}  ← A:2 > A:1 → A가 최신

동시 쓰기 (인과 관계 불명확):
  클라이언트 X → Node A: put("val", "10")  vc={A:2, B:0}
  클라이언트 Y → Node B: put("val", "20")  vc={A:0, B:1}

  비교:
    vc_A = {A:2, B:0}
    vc_B = {A:0, B:1}
    → A:2 > A:0 이지만 B:0 < B:1
    → 어느 쪽도 상대방을 완전히 포함하지 않음
    → 충돌 (Conflict) 확정

충돌 해결 옵션:
  1) 두 값 모두 보관 후 클라이언트에 전달 → 애플리케이션 레벨 결정
  2) 타임스탬프 기준 하나 선택 (LWW 폴백)
```

### 5.3 벡터 클록 판별 알고리즘

```
두 버전 vc_a, vc_b 비교:

함수 isDescendant(vc_a, vc_b):
  모든 노드 n에 대해 vc_a[n] >= vc_b[n] 이면 true
  (vc_a가 vc_b의 모든 이벤트를 포함하거나 초과 → vc_a가 최신)

판별 결과:
  isDescendant(vc_a, vc_b) = true  → vc_a가 최신
  isDescendant(vc_b, vc_a) = true  → vc_b가 최신
  둘 다 false                      → 동시 쓰기 → 충돌
  vc_a == vc_b                     → 동일한 버전
```

### 5.4 일관성 수준별 충돌 해결 정책 매핑

| 일관성 수준 | 충돌 해결 방식 | 이유 |
|-----------|-------------|------|
| `strong` | 충돌 자체가 발생하지 않음 | 쿼럼(W=N, R=N)으로 동시 쓰기 차단 |
| `eventual` | LWW (버전 번호 기준) + Read Repair | 실용적 균형 |
| `weak` | LWW (타임스탬프 기준) | 성능 우선, 단순 처리 |

## 6. 일관성 수준별 시나리오 비교

### 시나리오: 은행 잔액 차감

```
초기값: balance = 1000

클라이언트 A: get("balance") → 1000 읽음
클라이언트 B: get("balance") → 1000 읽음 (동시에)
클라이언트 A: put("balance", "800")  ← 200 차감
클라이언트 B: put("balance", "700")  ← 300 차감 (1000 기준으로)

기대값: 1000 - 200 - 300 = 500

weak(W=1,R=1):    최종값 700 또는 800 → 500이 아님 ❌ (부적합)
eventual(W=2,R=2): 최종값 700 또는 800 → 500이 아님 ❌ (부적합)
strong(W=3,R=3):  직렬화 보장으로 500에 가깝지만
                  진정한 원자적 연산은 별도 트랜잭션 필요 ⚠️

→ 금융 데이터는 strong 수준 + 애플리케이션 레벨 낙관적 잠금 병행 권장
```

### 시나리오: 사용자 프로필 업데이트

```
초기값: profile:1001 = { name: "Alice" }

클라이언트: put("profile:1001", { name: "Alice Kim" })

weak:     즉시 성공, 일부 노드에서 잠시 "Alice" 읽힐 수 있음 ⚠️ (허용 가능)
eventual: 거의 즉시 최신 값 반환 ✅ (적합)
strong:   전체 노드 동기화 후 응답, 약간 느림 (과한 수준)

→ 프로필 업데이트는 eventual 권장
```

### 시나리오: 세션 토큰 저장

```
로그인 시: put("session:abc123", "user:1001", ttl=3600)

weak:     즉시 저장, 일부 노드 미반영 시 재로그인 요구될 수 있음 ⚠️
eventual: 충분히 빠르게 전파됨 ✅
strong:   불필요하게 느림 (과한 수준)

→ 세션은 weak 또는 eventual 권장
```

## 7. 단조 읽기 보장 (Monotonic Read)

같은 클라이언트가 연속으로 읽을 때 시간이 거꾸로 가는 현상을 방지한다.

```
문제 시나리오:
  t=1: 클라이언트 → Node B: get("x") → version=5 ("Alice3") 읽음
  t=2: 클라이언트 → Node C: get("x") → version=3 ("Alice")  읽음 ← 과거로 역행!

  (Node C가 아직 복제를 받지 못한 상태)

해결 방법: 클라이언트 측 버전 추적
  - 클라이언트가 읽은 version 번호를 로컬에 기억
  - 다음 읽기 시 응답 version이 이전보다 낮으면
    → 다른 노드에 재요청하거나
    → 더 높은 version이 반영될 때까지 대기

  응답 헤더 X-Version을 클라이언트가 추적하여 구현
```

## 8. 쓰기 후 읽기 일관성 (Read-Your-Writes)

자신이 방금 쓴 값을 즉시 읽을 수 있어야 한다.

```
문제 시나리오:
  t=1: 클라이언트 A: put("profile:1001", "Alice Kim") → Node B에 저장
  t=2: 클라이언트 A: get("profile:1001")  → Node D 조회 (아직 복제 전)
       → "Alice" 반환 ← 방금 내가 쓴 값이 아님!

해결 방법:
  1) Strong 일관성 사용 → 쓰기 시 모든 노드 동기화 후 응답
  2) 클라이언트 Sticky Session → 같은 노드에만 읽기 요청
  3) 쓰기 후 version 번호를 전달 → 읽기 시 해당 version 이상 보장 요청

본 시스템 구현:
  쓰기 응답의 X-Version 헤더를 후속 GET 요청의
  X-Min-Version 헤더로 전달 (선택적)

  PUT /v1/keys/profile:1001
  → 응답: X-Version: 7

  GET /v1/keys/profile:1001
      X-Min-Version: 7
  → 코디네이터가 version ≥ 7인 노드에서만 읽기 수행
```

## 9. 일관성 수준 선택 가이드

```
데이터 손실이 절대 허용 안 됨?
  YES → strong

동시에 여러 클라이언트가 같은 키를 수정함?
  YES + 손실 불가 → strong
  YES + 손실 허용 → eventual

방금 쓴 값을 즉시 읽어야 함?
  YES → strong 또는 X-Min-Version 헤더 활용

일시적으로 오래된 값이 반환되어도 됨?
  YES + 성능 중요 → weak
  YES + 일반적 → eventual

캐시, 세션, 임시 데이터?
  → weak
```

## 10. 설계 파라미터 요약

| 파라미터 | 값 | 근거 |
|---------|-----|------|
| 기본 일관성 수준 | `eventual` (W=2, R=2) | 성능과 일관성의 균형 |
| 강한 일관성 | W=3, R=3 | 모든 노드 참여, 충돌 원천 차단 |
| 약한 일관성 | W=1, R=1 | 최고 성능, 손실 허용 |
| 버전 관리 방식 | 단조 증가 int64 | 단순하고 비교 비용 없음 |
| 충돌 감지 | 벡터 클록 | 동시 쓰기 정확히 판별 |
| 기본 충돌 해결 | LWW (버전 번호 기준) | 구현 단순성과 성능 |
| 단조 읽기 보장 | X-Version 헤더 추적 | 클라이언트 측 구현 |
| 쓰기 후 읽기 | X-Min-Version 헤더 | 선택적 적용 |
