```
- 설정된 처리율을 초과하는 요청은 정확하게 제한한다.
- 낮은 응답시간 : 이 처리율 제한 장치는 HTTP 응답시간에 나쁜 영향을 주어서는 곤란하다.
- 가능한 한 적은 메모리를 써야 한다.
- 분산형 처리율 제한(distributed rate limiting) : 하나의 처리율 제한 장치를 여러 서버나 프로세스에서 공유 할 수 있어야 한다.
- 예외 처리: 요청이 제한되었을 때는 그 사실을 사용자에게 분명하게 보여주어야 한다.
- 높은 결함 감내성(fault tolerance): 제한 장치에 장애가 생기더라도 전체 시스템에 영향을 주어서는 안된다.
```

## 요구사항 상세 정리

1. 설정된 처리율을 초과하는 요청은 정확하게 제한한다.
    - 정확하게 제한한다 -> 강한 일관성을 가진다.
        - + 4. 분산형 처리율 제한 - 하나의 처리율을 제한 장치를 여러 서버나 프로세스에서 공유 가능
        - 이를 보고, 많은 요청이 동시에 들어옴에도, 이를 동시성 이슈 없이 잘 처리할 수 있어야 한다.
2. 낮은 응답시간
    - 낮은 응답시간을 필요로 하기에, 처리율을 계산 하기 위해 저장되어야 하는 데이터가 Disk 보다는 메모리에 적재하는 방식이 옳다고 생각했음.
3. 예외 처리
    - 요청이 제한되면, 이를 사용자에게 에러를 Return 해줘야 하는 구나.라고만 생각함.
4. 높은 결함 감내성
    - 제한 장치가 여러 개가 있어서, 하나의 제한 장치가 꺼져도, 전체 시스템에는 영향이 없어야 한다.
    - 즉, 제한 장치가 다중화가 가능해야 한다.

1. 첫번째로 어떠한 저장소에 들어온 값들을 저장할 것인가 ?
    - 결국은 현재 실행 중인 갯수 및 최대 처리량/ 현재 처리량에 대한 데이터를 저장하고 있어야 한다.
    - 이 때 저장소는 처리가 무조건적으로 빨라야 한다.
    - Read-Modify-Write 가 빨라야 한다.
    - 정말로 Redis 뿐인가 ? 다른 것들은 시도 할 수 없는 가 ?
    - 다른 NoSql 을 생각해봤지만, Redis 만큼의 `낮은 응답속도` 를 만족 시키는 종류는 없다고 생각함.
2. 요청은 정확하게 처리하고, 분산형 처리율 제한을 위해 어떻게 처리할 수 있는 가 ?
    - 동시성 처리를 어떻게 정확하게 할 수 있을까 ?
    - 분산락
    - Lua Script 를 통한, RMW ( Read-Modify-Write ) 원자화를 통해 해결한다 .
3. 높은 결함 감내성 ( 고가용성, High Availability )
    - Redis 와 Gateway 서버 모두 다중화 구현(Scale Out) 이 가능한 가 ?
        - 그렇게 하게 됐을 경우에 문제는 없는 가 ?
    - Redis 의 경우, Sentinel 방식 또는 Cluster 방식으로 Scale Out 이 가능하다.
        - TPS 에 따라서, 부하가 심하지 않은 상황에서는 Sentinel 방식을  이용
        - 부하가 심해짐에 따라서, Cluster 방식으로 변경 필요
    - 게이트웨이는 카운터 상태를 Redis로 분리해 **무상태(stateless)** 로 설계
    - 로드밸런서(ALB)를 배치해 트래픽을 여러 게이트웨이 인스턴스로 분산한다

```markdown
| 구분 | Redis Sentinel | Redis Cluster |
|---|---|---|
| 핵심 목적 | 고가용성(HA) | 수평 확장(Scale-out) |
| 데이터 분산 | 없음 (단일 master) | 16384 슬롯 샤딩 (다중 master) |
| 쓰기 확장 | 불가 (단일 master 병목) | 가능 (샤드별 master) |
| 장애 조치 | Sentinel 정족수로 replica 승격 | 클러스터 자체(gossip)로 샤드별 승격 |
| 다중 키·Lua | 제약 없음 (전 데이터 한 노드) | 같은 슬롯만 (hash tag 필요, 아니면 CROSSSLOT) |
| 운영 복잡도 | 낮음 (최소: master1+replica+Sentinel3) | 높음 (최소 ≈6노드, 리샤딩·슬롯 이전) 
```

4. 어떻게 값을 제어하면서 처리율을 제한 할 것인 가 ?

- 단순히, 최대 실행 값을 놓고,  이를 현재 실행 수와 비교하면 진행할 것인가 ?
- 동시성 제한 과 처리율 제한
- **동시성 제한(concurrency limit)** = *지금 동시에 떠 있는* 수 (예: 동시 50개). 시작에 +1, 종료에 −1, 꽉 차면 대기/큐. **시간 개념이 없어요** — "현재 몇 개가 돌고 있냐"만 봅니다.
- **처리율 제한(rate limit)** = *단위 시간당* 요청 수 (예: 초당 100건). 토큰버킷·슬라이딩 윈도 같은 알고리즘이 여기 붙고, 카운터가 **시간에 따라** 리필/리셋됩니다. 핵심은 "시간"
- 가장 먼저 생각이 든 방식은
    - 11:01:00 ~ 11:01:59(1분) 사이에는 무조건적으로 특정 갯수의 API 요청만 인입 되도록 설정한다.
    - (고정 윈도우) 방식
    - 이렇게 진행했을 때의 단점으로는 경계 지점에 몰리게 되면, 특정 경우에는 인입이 가능해지고, 특정 경우에는 불가능 해지는 경우가 발생할 수 있다.
    - 그러하여, 슬라이딩 윈도우 방식을 고려함
    - 11:01:10 ~ 11:02:09 , 11:01:30 ~ 11:02:29 다음과 같이 , 시간이 지나감에 따라 윈도우가 이동하도록 하여 계산을 진행한다 .
    - Zset 자료 구조를  이용한다.
    - **ZREMRANGEBYSCORE key 0 (now−1분) - 윈도우에서 벗어나는 것들을 삭제한다.**
    - **ZCARD key - 남은 멤버수 계산**
    - **값에 따라서, ZADD 하여, 인입 결정**
