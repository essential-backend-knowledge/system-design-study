# 키-값 저장소 (Key-Value Store) 복제 전략 설계서

> **문서 목적**: 고가용성(99.9%) 요구사항을 충족하기 위해 데이터를 여러 노드에 복제하는 전략과 복제본 간 일관성 유지 방법을 상세히 정의한다.  
> **참조 문서**: `key-value-store-architecture.md`, `key-value-store-partitioning.md`, `key-value-store-data-model.md`  
> **핵심 결정**: 비동기 복제 + 쿼럼(N/W/R) 기반 일관성 수준 조정

---

## 1. 복제가 필요한 이유

파티셔닝만으로는 고가용성을 보장할 수 없다. 데이터가 한 노드에만 존재하면, 그 노드가 장애를 겪는 순간 데이터를 잃는다.

```
복제 없는 경우                  복제 있는 경우 (N=3)

  "user:1001" → Node A          "user:1001" → Node A (Primary)
                                              Node B (Replica 1)
                                              Node C (Replica 2)

  Node A 장애 →                  Node A 장애 →
  "user:1001" 접근 불가 ❌        Node B 또는 C에서 서비스 유지 ✅
  데이터 영구 손실 가능 ❌        데이터 보존 ✅
```

---

## 2. 복제 파라미터 정의

### 2.1 N / W / R 쿼럼

복제 동작은 세 가지 값으로 제어한다.

| 파라미터 | 의미 | 본 시스템 기본값 |
|---------|------|--------------|
| **N** | 복제본 총 수 (몇 개 노드에 저장할지) | 3 |
| **W** | 쓰기 성공으로 인정하기 위한 최소 응답 수 | 2 |
| **R** | 읽기 성공으로 인정하기 위한 최소 응답 수 | 2 |

N=3, W=2, R=2가 기본값인 이유:

```
W + R = 4 > N = 3

→ W + R > N 조건 충족
→ 읽기 집합과 쓰기 집합이 반드시 1개 이상 겹침
→ 항상 최신 데이터를 포함한 노드가 읽기에 참여
→ 강한 일관성에 가까운 동작 보장
```

### 2.2 쿼럼 조합별 특성

API 명세서의 `X-Consistency-Level` 헤더가 내부적으로 아래 쿼럼 값과 매핑된다.

| 일관성 수준 | W | R | 특성 | 적합한 용도 |
|-----------|---|---|------|-----------|
| `strong` | 3 | 3 | 모든 노드 동의. 가장 안전하나 느림 | 결제, 잔액, 재고 |
| `eventual` (기본) | 2 | 2 | 균형잡힌 성능과 일관성 | 일반 서비스 |
| `weak` | 1 | 1 | 최고 성능. 일시적으로 오래된 값 가능 | 캐시, 세션, 로그 |

## 3. 복제본 배치 전략

### 3.1 해시 링 기반 복제본 위치 결정

파티셔닝 설계서에서 정의한 해시 링을 그대로 활용한다. 키의 Primary 노드로부터 **시계 방향으로 N-1개의 다음 노드**가 복제본을 담당한다.

```
해시 링 위의 노드 순서 (시계 방향):
  Node A → Node B → Node C → Node D → (다시 Node A...)

put("user:1001") → 해시 → Node B가 Primary

  복제본 배치 (N=3):
    Node B: Primary   (1번째)
    Node C: Replica 1 (2번째, 시계 방향)
    Node D: Replica 2 (3번째, 시계 방향)

  Node A는 이 키와 무관 (담당 범위 밖)
```

### 3.2 물리 노드 중복 방지

가상 노드(Virtual Node) 환경에서는 시계 방향으로 탐색할 때 **이미 선택된 물리 노드는 건너뜀**으로써 복제본이 서로 다른 물리 노드에 배치되도록 보장한다.

```
가상 노드 탐색 시 중복 방지 예시

  시계 방향 탐색 순서:
    vNode-B#12 → Node B  ← Primary 선택
    vNode-B#43 → Node B  ← 이미 선택됨, 건너뜀
    vNode-C#7  → Node C  ← Replica 1 선택
    vNode-D#31 → Node D  ← Replica 2 선택

  결과: Node B, C, D 각각 1개씩 → 서로 다른 물리 서버에 복제 보장
```

### 3.3 랙(Rack) 인식 배치

데이터센터 환경에서는 같은 랙의 서버들이 동시에 장애를 겪을 수 있다 (전원 장치, 스위치 공유). 복제본을 서로 다른 랙에 배치하여 이를 방지한다.

```
랙 구성 예시:
  Rack 1: Node A, Node B
  Rack 2: Node C, Node D
  Rack 3: Node E, Node F

복제본 배치 규칙 (N=3):
  Primary → Rack 1의 Node A
  Replica 1 → Rack 1에 이미 있으므로 → Rack 2의 Node C
  Replica 2 → Rack 1, 2에 이미 있으므로 → Rack 3의 Node E

→ 랙 하나가 통째로 장애나도 나머지 2개 복제본으로 서비스 유지
```

## 4. 쓰기 복제 흐름

### 4.1 정상 쓰기 흐름 (W=2, N=3)

```
① 클라이언트 → 코디네이터: PUT user:1001 = "Alice"

② 코디네이터: 해시 링에서 담당 노드 결정
              Primary = Node B
              Replica = Node C, Node D

③ 코디네이터 → Node B, C, D 동시 쓰기 요청 (병렬)

④ 각 노드 내부 처리:
     WAL에 순차 기록 (내구성 확보)
         ↓
     Memtable에 저장 (빠른 응답)
         ↓
     코디네이터에 성공 응답

⑤ 코디네이터: W=2개 응답 수신 시점에 클라이언트에 성공 반환
              (3번째 노드 응답은 백그라운드에서 계속 대기)

타임라인:
  t=0ms   코디네이터 → Node B, C, D 동시 요청
  t=3ms   Node B 성공 응답  (1/2)
  t=5ms   Node C 성공 응답  (2/2) ← W=2 충족, 클라이언트에 응답
  t=8ms   Node D 성공 응답  (3/3, 백그라운드)
```

### 4.2 일부 노드 지연 시 쓰기 흐름

```
Node D가 응답이 느릴 때 (네트워크 지연)

  t=0ms   코디네이터 → Node B, C, D 동시 요청
  t=3ms   Node B 성공 (1/2)
  t=5ms   Node C 성공 (2/2) ← W=2 충족, 클라이언트에 즉시 응답
  t=300ms Node D 성공 (뒤늦게 완료, 백그라운드)

  → 클라이언트는 5ms 시점에 응답받음
  → Node D의 지연이 클라이언트 응답 시간에 영향 없음
```

### 4.3 노드 장애 시 쓰기 흐름 (Hinted Handoff)

```
Node D가 장애 상태일 때

① 코디네이터 → Node B, C 쓰기 성공 (W=2 충족)
② 코디네이터 → Node D 쓰기 실패

③ 코디네이터가 Node D 대신 Node E(다음 시계 방향 노드)에
   "힌트(Hint)"와 함께 임시 저장:

   힌트 내용: {
     "intended_node": "Node D",
     "key":           "user:1001",
     "value":         "Alice",
     "version":       5,
     "stored_at":     "2026-06-03T14:00:00Z"
   }

④ Node D 복구 감지 (가십 프로토콜)
⑤ Node E → Node D: 힌트 데이터 전달
⑥ Node D 저장 완료 → Node E의 힌트 데이터 삭제
```

힌트 보존 기간: 최대 **24시간**. 이후에도 복구되지 않으면 힌트 폐기 (Anti-Entropy로 보완).

## 5. 읽기 복제 흐름

### 5.1 정상 읽기 흐름 (R=2, N=3)

```
① 클라이언트 → 코디네이터: GET user:1001

② 코디네이터: 해시 링에서 담당 노드 결정
              Node B (Primary), Node C, Node D

③ 코디네이터 → Node B, C, D 동시 읽기 요청 (병렬)

④ 각 노드: 블룸 필터 → Memtable → SSTable 순서로 탐색
           값 + version 번호 반환

⑤ 코디네이터: R=2개 응답 수신
              version 번호가 가장 높은 값을 최신으로 선택

  응답 예시:
    Node B: value="Alice", version=5
    Node C: value="Alice", version=5  ← R=2 충족, 일치
    Node D: (응답 대기 중)

⑥ 클라이언트에 value="Alice" 반환
```

### 5.2 복제본 간 값이 다를 때 (버전 불일치)

```
네트워크 파티션 후 복구 직후 상황

  Node B: value="Alice", version=5  (최신)
  Node C: value="Anna",  version=3  (오래된 값)

코디네이터:
  version 5가 더 높음 → "Alice" 선택하여 클라이언트에 반환

  ↓ Read Repair 수행 (백그라운드)

코디네이터 → Node C: version=5인 "Alice"로 업데이트 요청
Node C: 업데이트 완료 → 이후 읽기에서도 "Alice" 반환
```

## 6. Read Repair (읽기 시 복구)

읽기 요청을 처리하는 과정에서 복제본 간 불일치를 자동으로 감지하고 수정한다.

### 6.1 동작 방식

```
읽기 응답 수집 단계에서:

  모든 N개 노드의 응답 version을 비교
  max_version = 응답 중 가장 높은 version

  for each node in responses:
    if node.version < max_version:
      백그라운드로 해당 노드에 최신 값 동기화 요청
      (클라이언트 응답 지연 없음)
```

### 6.2 Read Repair 적용 범위

| 일관성 수준 | Read Repair 적용 여부 |
|-----------|-------------------|
| `strong` | ✅ 항상 적용 (동기) |
| `eventual` | ✅ 적용 (비동기) |
| `weak` | ❌ 미적용 (성능 우선) |

---

## 7. Anti-Entropy (백그라운드 동기화)

Read Repair는 읽기 요청이 있을 때만 불일치를 발견한다. 오랫동안 읽히지 않는 데이터의 불일치는 Anti-Entropy 프로세스가 주기적으로 감지하고 수정한다.

### 7.1 Merkle Tree 기반 비교

전체 데이터를 일일이 비교하는 것은 비용이 크다. Merkle Tree를 사용하면 다른 구간만 빠르게 찾을 수 있다.

```
Merkle Tree 구조 (각 노드는 자식 해시의 합성 해시)

            [Root Hash]
           /            \
    [Hash A-B]        [Hash C-D]
    /       \          /       \
[Hash A] [Hash B] [Hash C] [Hash D]
   |         |       |         |
 키 범위A  키 범위B  키 범위C  키 범위D

두 노드 간 비교:
  Root Hash가 같으면 → 전체 데이터 일치, 비교 종료
  Root Hash가 다르면 → 하위로 내려가 다른 구간 탐색

  → 전체 데이터 전송 없이 불일치 구간만 특정 가능
  → O(log N) 비교로 불일치 위치 탐색
```

### 7.2 Anti-Entropy 실행 주기

| 항목 | 값 |
|------|-----|
| 실행 주기 | 1시간마다 (백그라운드) |
| 비교 단위 | 노드 쌍 (Primary ↔ 각 Replica) |
| 불일치 해결 | 더 높은 version의 값으로 덮어씀 |
| 실행 시간대 | 트래픽이 적은 새벽 시간대 집중 실행 |

## 8. 복제 지연 모니터링

복제본 간 데이터 차이가 얼마나 지속되는지 추적하는 지표다.

| 지표 | 설명 | 경보 기준 |
|------|------|---------|
| **복제 지연 (Replication Lag)** | Primary 쓰기 후 Replica 반영까지 걸린 시간 | 평균 500ms 초과 시 |
| **불일치 키 수** | Anti-Entropy 실행 시 발견된 불일치 키 개수 | 1,000개 초과 시 |
| **Hinted Handoff 대기 수** | 전달되지 못한 힌트 데이터 개수 | 10,000개 초과 시 |
| **Read Repair 발생율** | 전체 읽기 중 Read Repair가 수행된 비율 | 5% 초과 시 |

## 9. 복제 설계 파라미터 요약

| 파라미터 | 값 | 근거 |
|---------|-----|------|
| N (복제본 수) | 3 | 노드 1개 장애에도 과반수(2개) 유지 |
| W (쓰기 쿼럼, 기본) | 2 | W+R>N 충족하면서 쓰기 성능 확보 |
| R (읽기 쿼럼, 기본) | 2 | W+R>N 충족하면서 읽기 성능 확보 |
| 복제 방식 | 비동기 병렬 | 응답 지연 최소화 |
| 복제본 위치 결정 | 해시 링 시계 방향 N개 노드 | 파티셔닝과 일관된 방식 |
| 랙 인식 배치 | 서로 다른 랙에 복제본 분산 | 랙 단위 장애 대응 |
| Hinted Handoff 보존 | 최대 24시간 | 단기 장애 복구 대응 |
| Read Repair | 비동기 (weak 제외) | 클라이언트 응답 지연 없이 동기화 |
| Anti-Entropy 주기 | 1시간 | 장기 불일치 감지 및 수정 |
| 불일치 비교 방식 | Merkle Tree | O(log N) 효율적 비교 |
| 충돌 해결 기본값 | Last-Write-Wins (version 기준) | 구현 단순성과 성능 |
