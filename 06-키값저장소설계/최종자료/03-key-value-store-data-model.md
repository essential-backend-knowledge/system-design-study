# 키-값 저장소 (Key-Value Store) 데이터 모델 설계서

> **문서 목적**: 키-값 저장소에서 데이터가 어떤 구조로 정의되고, 저장되고, 관리되는지를 명세한다.  
> **참조 문서**: `key-value-store-requirements.md`, `key-value-store-architecture.md`  
> **핵심 제약**: 키-값 쌍 하나의 크기는 10KB 이하

---

## 1. 데이터 단위 정의

### 1.1 키-값 쌍 (Key-Value Pair)

시스템에서 저장과 조회의 기본 단위는 **키-값 쌍**이다.

```
┌─────────────────────────────────────────────────────┐
│                    키-값 쌍 (Entry)                   │
│                                                     │
│  ┌─────────────┐        ┌──────────────────────┐    │
│  │    Key      │ ──────▶│       Value          │    │
│  │  (식별자)   │        │      (데이터)         │    │
│  └─────────────┘        └──────────────────────┘    │
│                                                     │
│  + Metadata (내부 관리용)                             │
│    - version    : 버전 번호 (충돌 해결용)              │
│    - timestamp  : 마지막 쓰기 시각                    │
│    - ttl        : 만료 시각 (선택)                    │
│    - tombstone  : 삭제 여부 마커                      │
└─────────────────────────────────────────────────────┘
```

---

## 2. 키 (Key) 모델

### 2.1 키의 구조

키는 바이트 배열(byte array)로 취급한다. 형태에 제한은 없으나, 성능과 가독성을 위해 아래 규칙을 권장한다.

```
권장 키 형식:  {namespace}:{entity}:{id}

예시:
  user:profile:1001       → 사용자 1001의 프로필
  session:token:abc123    → 세션 토큰
  product:stock:item-99   → 상품 재고
  cache:result:253DDEC4   → 해시 기반 캐시 키
```

### 2.2 키 제약 조건

| 항목 | 규칙 |
|------|------|
| **유일성** | 저장소 전체에서 반드시 고유해야 함 |
| **형태** | 일반 텍스트 또는 해시 값 모두 허용 |
| **최대 크기** | 키 + 값 합산 10KB 이하 (키는 가능한 짧게) |
| **문자 제한** | 제어 문자(null, newline 등) 포함 불가 |
| **대소문자** | 구분함 (`User:1` ≠ `user:1`) |
| **접근 방식** | 키를 통해서만 값에 접근 가능. 범위 쿼리 미지원 |

### 2.3 키 타입별 비교

| 키 타입 | 예시 | 장점 | 단점 |
|--------|------|------|------|
| **텍스트 키** | `last_logged_in_at` | 가독성 높음, 디버깅 용이 | 길이가 길어질 수 있음 |
| **해시 키** | `253DDEC4` | 고정 길이, 분산 균일 | 가독성 없음 |
| **복합 키** | `user:profile:1001` | 네임스페이스 구분 가능 | 파싱 규칙 필요 |

## 3. 값 (Value) 모델

### 3.1 값의 특성

저장소는 값의 내부 구조를 해석하지 않는다 (Opaque Value). 즉, 저장소 입장에서 값은 단순한 **바이트 배열**이다.

```
저장소가 아는 것:
  - 값의 크기 (bytes)
  - 값의 버전 (version)

저장소가 모르는 것:
  - 값이 문자열인지, JSON인지, 바이너리인지
  - 값의 내부 필드 구조
  - 값의 의미
```

### 3.2 값으로 허용되는 타입

클라이언트가 직렬화(Serialization)하여 저장할 수 있는 대표 형태는 다음과 같다.

| 타입 | 예시 | 비고 |
|------|------|------|
| **문자열** | `"Alice"` | 가장 단순한 형태 |
| **정수 / 실수** | `42`, `3.14` | 문자열로 직렬화하여 저장 |
| **JSON 객체** | `{"name":"Alice","age":30}` | 구조화 데이터에 적합 |
| **리스트** | `["read","write","admin"]` | JSON 배열로 직렬화 |
| **바이너리** | `\x89PNG\r\n...` | 이미지, 파일 등 |

### 3.3 값 크기 제약

```
키 크기 + 값 크기 ≤ 10KB (10,240 bytes)

예시:
  키  "user:profile:1001"  = 18 bytes
  값  { JSON 객체 }        = 최대 10,222 bytes 까지 허용
```

10KB를 초과하는 데이터가 필요한 경우 클라이언트 측에서 분할 저장해야 한다.

```
큰 데이터 분할 저장 패턴:
  file:chunk:abc123:0  →  [0~10KB 데이터]
  file:chunk:abc123:1  →  [10~20KB 데이터]
  file:chunk:abc123:2  →  [20~30KB 데이터]
  file:meta:abc123     →  { "chunks": 3, "size": 28500 }
```

## 4. 메타데이터 (Metadata) 모델

### 4.1 내부 저장 포맷

클라이언트에게는 보이지 않지만, 내부적으로 각 키-값 쌍은 메타데이터와 함께 저장된다.

```
내부 저장 단위 (Internal Entry)

┌──────────────────────────────────────────────────┐
│  key        : bytes      키 원본                  │
│  value      : bytes      값 원본                  │
│  version    : int64      단조 증가 버전 번호        │
│  timestamp  : int64      Unix 타임스탬프 (ms)      │
│  ttl        : int64      만료 시각 (0 = 영구 보존) │
│  tombstone  : bool       삭제 마커 (true = 삭제됨) │
│  checksum   : uint32     데이터 무결성 검증용 CRC32 │
└──────────────────────────────────────────────────┘
```

### 4.2 메타데이터 필드 상세

#### version (버전 번호)

복제 환경에서 여러 노드의 값이 충돌할 때, 어느 값이 최신인지 판단하는 기준이다.

```
시나리오: 네트워크 파티션 후 복구

Node A: user:1001 → "Alice"  (version: 5)
Node B: user:1001 → "Bob"    (version: 7)  ← 더 높은 버전이 최신
Node C: user:1001 → "Alice"  (version: 5)

복구 후: version 7인 "Bob"으로 통일 (Last-Write-Wins)
```

#### tombstone (삭제 마커)

키-값을 물리적으로 즉시 삭제하면 복제본 간 동기화 문제가 발생한다. 대신 삭제를 논리적으로 표시하고, 나중에 Compaction 시 실제 제거한다.

```
delete("user:1001") 호출 시:

  실제 동작:
  key       : "user:1001"
  value     : (비어있음)
  tombstone : true         ← 삭제됨을 표시
  version   : 8

  Compaction 전까지는 디스크에 존재하지만
  get("user:1001") 호출 시 "없음"으로 응답
```

#### ttl (Time-To-Live)

만료 시각을 설정하면, 해당 시각 이후 get() 요청에서 "없음"으로 응답한다. 세션, 캐시 등에 활용한다.

```
put("session:abc123", "user:1001", ttl=3600)
  → 현재 시각 + 3600초 후 자동 만료

만료 후 get("session:abc123")
  → null 반환 (내부적으로는 tombstone 처리)
```

#### checksum (체크섬)

저장 및 읽기 시점에 CRC32 체크섬을 계산하여 데이터 손상(bit rot)을 감지한다.

```
쓰기 시: checksum = CRC32(key + value) 계산 후 함께 저장
읽기 시: CRC32(key + value) 재계산 → 저장된 checksum과 비교
         불일치 시 → 다른 복제본에서 재시도
```

## 5. 저장소 계층별 데이터 표현

### 5.1 Memtable (메모리)

메모리 내에서는 정렬된 트리 구조(Red-Black Tree 또는 Skip List)로 관리한다.

```
Memtable (정렬된 순서로 유지)

  "product:stock:item-10"  →  { value: "50",  version: 3 }
  "product:stock:item-99"  →  { value: "120", version: 1 }
  "session:abc123"         →  { value: "user:1001", version: 2, ttl: 1719820800 }
  "user:profile:1001"      →  { value: "{...}", version: 7 }
  "user:profile:2345"      →  { value: "{...}", version: 4 }
```

정렬을 유지하는 이유: SSTable로 플러시할 때 순서대로 기록하면 디스크 순차 쓰기가 가능하여 성능이 좋다.

### 5.2 SSTable (디스크)

디스크에는 정렬된 불변(Immutable) 파일 형태로 저장된다. 각 SSTable 파일은 인덱스 블록과 데이터 블록으로 구성된다.

```
SSTable 파일 구조

┌──────────────────────────────────────┐
│            인덱스 블록                │  ← 각 키의 파일 내 오프셋 저장
│  "product:stock:item-10" → offset: 0 │     (빠른 탐색용)
│  "session:abc123"        → offset: 88│
│  "user:profile:1001"     → offset:150│
├──────────────────────────────────────┤
│            데이터 블록                │  ← 실제 키-값 + 메타데이터
│  [key_len][key][value_len][value]     │
│  [version][timestamp][ttl][tombstone] │
│  [checksum]                           │
│  ...반복...                           │
├──────────────────────────────────────┤
│           푸터 (Footer)              │  ← 인덱스 블록 시작 오프셋
└──────────────────────────────────────┘
```

### 5.3 WAL (Write-Ahead Log)

쓰기 요청이 들어오면 Memtable에 반영하기 전에 WAL에 순차 기록한다. 장애로 Memtable이 유실되면 WAL을 재생(replay)하여 복구한다.

```
WAL 레코드 형식 (한 줄 = 한 쓰기 요청)

  [seq_no] [timestamp] [op] [key] [value] [checksum]

예시:
  0001  1719820000  PUT  user:profile:1001  {...}  A3F2
  0002  1719820001  PUT  session:abc123     user:1001  B1C4
  0003  1719820005  DEL  session:xyz999     -       D9E1
```

## 6. 버전 충돌 해결 정책

### 6.1 Last-Write-Wins (LWW)

가장 단순한 정책. 타임스탬프 또는 버전 번호가 높은 값을 최신으로 채택한다.

```
장점: 구현이 단순하고 성능이 좋음
단점: 동시 쓰기 시 한쪽 데이터가 유실될 수 있음
적합: 캐시, 세션 등 손실이 허용되는 데이터
```

### 6.2 벡터 클록 (Vector Clock)

노드별 버전 번호를 배열로 관리하여, 인과 관계를 추적한다.

```
Node A: user:1001 → version: [A:3, B:1, C:2]
Node B: user:1001 → version: [A:3, B:2, C:2]  ← B만 더 높음 → B가 최신

인과 관계가 불명확할 경우 (충돌):
  Node A: [A:3, B:1]
  Node B: [A:2, B:2]  ← 어느 쪽도 상대방보다 크지 않음 → 충돌

충돌 해결: 클라이언트에게 두 값을 모두 전달하여 애플리케이션 레벨에서 결정
```

### 6.3 본 시스템의 기본 정책

| 일관성 수준 | 충돌 해결 정책 |
|-----------|-------------|
| 약한 일관성 | Last-Write-Wins (타임스탬프 기준) |
| 최종 일관성 | Last-Write-Wins (버전 번호 기준) |
| 강한 일관성 | 쿼럼(W+R>N)으로 충돌 자체를 방지 |

## 7. 데이터 생명주기

```
생성
  put(key, value) 호출
       │
       ▼
활성 상태
  get(key) 로 정상 조회 가능
  put(key, newValue) 로 버전 증가하며 갱신
       │
       ├──── ttl 만료 ──────────────────────────┐
       │                                        ▼
       │                               만료 상태 (Expired)
       │                               get() → null 반환
       │                               tombstone 처리 후
       │                               Compaction 시 제거
       │
       ├──── delete(key) 호출 ──────────────────┐
       │                                        ▼
       │                               삭제 마킹 (Tombstone)
       │                               논리적 삭제 상태
       │                               Compaction 시 물리 제거
       │
       ▼
  Compaction
  오래된 버전 / tombstone / 만료 항목 물리 제거
  SSTable 파일 병합 및 최적화
```

## 8. 데이터 모델 제약 요약

| 항목 | 규칙 |
|------|------|
| 키-값 쌍 최대 크기 | 10KB 이하 |
| 키 유일성 | 전역 유일 |
| 값 타입 제한 | 없음 (Opaque Bytes) |
| 범위 쿼리 | 미지원 (키를 통한 단건 접근만 가능) |
| 삭제 방식 | 논리적 삭제 (Tombstone) → Compaction 시 물리 제거 |
| 버전 관리 | 단조 증가 버전 번호 (int64) |
| 무결성 검증 | CRC32 체크섬 |
| 만료 정책 | TTL 지원 (선택적) |
