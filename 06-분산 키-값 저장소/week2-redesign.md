# 2주차 - 재설계

## 1. 기존 1차 설계에서 부족했던 점

1차 설계는 **데이터를 어떻게 나누고(파티션) · 어떻게 복제하고 · 어떻게 일관성을 조정하나**까지는 다뤘다. 하지만 "정상 동작할 때"의 그림이었고, **고장 났을 때**와 **노드 내부에서 실제로 어떻게 읽고 쓰나**가 비어 있었다.

| 1주차에서 다루지 못 한 주제 | 내용                                             |
|-----------------|------------------------------------------------|
| `장애 처리`         | 장애를 **감지**하는 법, 일시적/영구적 장애를 **나눠서** 다루는 법      |
| `쓰기/읽기 경로`      | 한 노드가 요청을 받아 **디스크에 어떻게 저장**하고 읽는지 |
| `충돌 해소의 한계`     | 벡터 시계만 적고, "그래서 충돌 나면 **누가** 병합하나"의 운영 부담을 안 따짐 |

→ **happy path는 설계했지만 failure path와 storage engine이 비어 있었다.**

---

## 2. 책을 읽고 새로 알게 된 핵심 포인트

### 2-1. 장애는 "감지 > 일시적 > 영구적" 3단계로 나눠 푼다

1차에서 "장애 처리"를 한 덩어리로 봤는데, 책에서는 **종류별로** 쪼갠다.

**(1) 장애 감지 — 가십 프로토콜(Gossip)**
- "한 노드가 죽었다"를 한 노드의 판단으로 정하지 않는다. (오탐 위험)
- 모든 노드가 멤버십 리스트(노드ID, heartbeat 카운터)를 주기적으로 **랜덤한 다른 노드와 교환**.
- 어떤 노드의 heartbeat가 한참 안 늘면 → 가십으로 퍼뜨려 **여러 노드가 합의**해야 "장애"로 본다.
- all-to-all 하트비트(모두가 모두에게)는 노드 수 제곱으로 트래픽이 늘어 못 씀 → 그래서 가십.

**(2) 일시적 장애 — 느슨한 정족수(Sloppy Quorum) + 단서 후 넘김(Hinted Handoff)**
- 노드가 잠깐 죽었다고 쓰기를 거부하면 가용성이 깨진다.
- W개를 **원래 담당 노드**에서만 받지 않고, 링에서 **살아있는 W개 노드**에서 받는다(느슨한 정족수).
- 원래 담당 노드가 죽어 대신 받은 노드는 "이건 원래 A 거였음"이라는 **단서(hint)**를 들고 임시 보관.
- A가 살아나면 -> 보관하던 데이터를 **A에게 넘겨줌(handoff)** -> 일관성 복구.

**(3) 영구적 장애 — 안티 엔트로피(Anti-entropy) + 머클 트리(Merkle Tree)**
- 노드가 영영 죽거나, 복제본끼리 데이터가 오래 어긋난 경우.
- 두 복제본의 **전체 데이터를 다 비교**하면 너무 비싸다.
- 머클 트리: 키 범위를 해시로 **트리화** -> 루트 해시만 비교, 다르면 **다른 가지로만 내려가며** 차이 난 부분만 동기화.
- -> 비교 비용이 전체가 아니라 **차이 난 만큼**으로 줄어든다.

**(4) 데이터센터 장애**
- 복제본 N개를 **서로 다른 데이터센터/리전**에 분산 → 한 DC가 통째로 나가도 서비스 유지.

### 2-2. 저장 엔진 — 쓰기/읽기 경로 (LSM 트리 계열)

카산드라/빅테이블 계열의 **LSM 트리** 구조다.

**쓰기 경로**
1. **커밋 로그(Commit Log)**에 먼저 추가 기록(append) -> 죽어도 복구 가능.
2. **멤테이블(MemTable, 메모리)**에 기록 -> 쓰기가 메모리 속도라 빠름.
3. 멤테이블이 임계치를 넘으면 디스크의 **SSTable**로 flush(정렬된 불변 파일).

**읽기 경로**
1. **멤테이블** 먼저 확인 -> 있으면 바로 반환.
2. 없으면 **블룸 필터(Bloom Filter)**로 "이 SSTable에 키가 있을 가능성"을 먼저 검사 -> 없는 게 확실한 파일은 디스크 안 읽음.
3. 가능성 있는 SSTable만 읽어 값 반환.
- 시간이 지나면 SSTable이 쌓이므로 **컴팩션(compaction)**으로 병합·정리.

핵심 통찰: **쓰기는 append-only라 빠르고, 읽기는 블룸 필터로 디스크 I/O를 쳐낸다.**

### 2-3. 일관성 모델을 명시적으로 구분
- **강한 일관성**: 읽으면 항상 최신값(쓰기 전엔 응답 보류) -> 가용성/지연 희생.
- **약한 일관성**: 최신값 못 읽을 수 있음.
- **최종 일관성**: 약한 일관성의 특수형. 갱신이 멈추면 결국 모든 복제본이 수렴.
- 우리 설계의 기본값은 **최종 일관성 + 정족수로 끌어올리기**임을 명확히.

## 3. 보완한 최종 아키텍처

| 영역 | 1차 | 2차(보완) |
|---|---|---|
| 파티션 | 일관성 해싱 + 가상 노드 | (유지) |
| 복제 | 시계방향 N개 노드 | + **DC/랙 분산 배치 명시** |
| 일관성 | 정족수 W/R/N | + **strong/weak/eventual 모델 구분** |
| 충돌 해소 | 벡터 시계 | (유지) + LWW 대안 검토(§우리 서비스) |
| **장애 감지** | ❌ | **가십 프로토콜** |
| **일시 장애** | ❌ | **느슨한 정족수 + 단서 후 넘김** |
| **영구 장애** | ❌ | **안티 엔트로피 + 머클 트리** |
| **DC 장애** | ❌ | **리전 간 복제** |
| **쓰기 경로** | ❌ | **커밋 로그 → 멤테이블 → SSTable** |
| **읽기 경로** | ❌ | **멤테이블 → 블룸 필터 → SSTable → 컴팩션** |

**요청 흐름(쓰기 기준)**
```
클라이언트
  -> (아무 노드나) 코디네이터 노드
  -> 일관성 해싱으로 담당 N개 노드 결정
  -> 살아있는 W개에 복제 쓰기 (죽었으면 느슨한 정족수 + hint)
  -> 각 노드: 커밋 로그 → 멤테이블 → (임계치) SSTable
  -> W개 ACK 모이면 성공 응답
  배경: 가십으로 멤버십 관리, 머클 트리로 복제본 정합성 복구
```

## 4. 다음 설계 때 가져갈 교훈

1. **설계는 happy path 뿐만 아니라 failure path를 잘 고려해야 한다.** 차별점은 "실패 시 무엇을 포기하고 무엇을 지키나"에서 나온다.
2. **'장애 처리' 같은 큰 덩어리는 종류별로 쪼갠다.** (감지 / 일시적 / 영구적 / DC) -> 한 번에 풀려 하지 말 것.
3. **저장 엔진(노드 내부)까지 내려가야 지연·처리량 근거가 생긴다.** "p99 < 10ms"를 주장하려면 append-only 쓰기·블룸 필터 같은 내부 메커니즘을 고려할 수 있어야 한다.
4. **추상적 용어("최종 일관성")는 한 단계 더 구체화**해서 적자. 모델 구분과 W/R/N 숫자까지 내려가야 면접/리뷰에서 안 막힌다.

## 1주차 vs 2주차

- **내 설계에서 놓친 것은 무엇인가?**
  - 장애 **감지** 메커니즘 자체가 없었다. "노드가 죽으면 복제본으로 버틴다"까지만 생각하고, *어떻게 죽은 걸 아는가*(가십)를 빼먹음.
  - 장애를 **일시적/영구적으로 구분**하지 않아 sloppy quorum·hinted handoff·머클 트리라는 표준 해법에 도달 못 함.
  - **저장 엔진**(쓰기/읽기 경로)을 통째로 비워서, 정작 핵심 요구사항인 *p99 < 10ms*의 근거를 못 댐.

- **책의 설계가 더 나은 이유는 무엇인가?**
  - 장애를 **종류별로 분해**해 각각에 맞는 표준 패턴을 매칭한다 -> 막연한 "장애 처리"가 실행 가능한 설계가 됨.
  - **메모리(멤테이블)–디스크(SSTable)–블룸 필터**로 쓰기/읽기 비용을 비대칭적으로 최적화 -> 성능 주장에 근거가 생김.
  - 가십·느슨한 정족수처럼 **중앙 조정자 없이도** 동작하는 패턴이라, 1차에서 고른 "분산형(SPOF 없음)" 결정과 일관되게 맞물림.

- **책의 설계도 실제 운영 환경에서는 어떤 한계가 있을까?**
  - **벡터 시계의 운영 부담**: 충돌 시 "병합"을 클라이언트에 떠넘긴다. 모든 클라이언트가 병합 로직을 올바로 구현해야 하고, 리스트가 길어지는 문제도 있음.
  - **느슨한 정족수는 일관성을 더 약하게** 만든다. "원래 담당 아닌 노드"가 받는 동안 `W+R>N`의 보장이 흔들릴 수 있음 -> 가용성을 위해 일관성을 더 양보한 것.
  - **컴팩션/안티 엔트로피의 비용**: 백그라운드 머클 트리 비교·SSTable 병합이 디스크 I/O·CPU를 잡아먹어 **p99 지연 스파이크**를 만든다(설계도엔 안 보이는 운영 현실).
  - **가십은 수렴이 느리다**: 장애 전파에 시간이 걸려, 감지 직후 짧은 구간엔 죽은 노드로 요청이 갈 수 있음.

- **우리 서비스(포트로지스 · 물류 도메인)라면 다르게 설계할 부분은 무엇인가?**
  - **충돌 해소: 벡터 시계 대신 LWW(Last-Write-Wins)가 더 맞을 수 있다.** 트럭/컨테이너 **실시간 위치**는 "합집합 병합"이 무의미하고 *가장 최근 좌표*가 곧 정답이다 -> 타임스탬프 기반 LWW가 단순하고 도메인에 부합. (단, 시계 동기화 이슈는 감수)
  - **워크로드가 쓰기 우위**: 위치/상태가 초 단위로 갱신되는 추적 데이터는 read-heavy 가정과 다름 -> LSM 트리(쓰기 친화적)는 오히려 잘 맞고, W 정족수를 낮춰 쓰기 지연을 줄이는 튜닝이 유효.
  - **데이터 성격 분리**: 위치/상태 같은 **추적성 데이터**는 이 KV 저장소(AP, 최종 일관성)에, **계약·정산·결제** 같은 데이터는 강한 일관성이 필요하니 KV가 아니라 **RDB(CP)**로 분리. "모든 걸 한 저장소에" 넣지 않기.
  - **파티션 키 설계에 지역성 반영**: 물류는 항만/리전 단위 조회가 잦음 -> 단순 `hash(key)`보다 **리전을 접두로 둔 키**로 지역 단위 질의·DC 배치를 정렬하면 지연·정합성에 유리.
  - **규모 재산정**: B2B 물류는 인터넷 서비스 대비 키/트래픽 규모가 작을 수 있음 -> "수조 키" 가정을 그대로 쓰지 말고 *실제 화물·차량 수* 기준으로 N·가상노드 수를 다시 잡기.
