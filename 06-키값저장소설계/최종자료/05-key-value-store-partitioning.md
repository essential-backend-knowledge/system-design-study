# 키-값 저장소 (Key-Value Store) 파티셔닝 전략 설계서

> **문서 목적**: 대용량 데이터를 여러 스토리지 노드에 균등하게 분산 저장하기 위한 파티셔닝 전략을 상세히 정의한다.  
> **참조 문서**: `key-value-store-requirements.md`, `key-value-store-architecture.md`  
> **핵심 결정**: 안정 해시(Consistent Hashing) + 가상 노드(Virtual Node) 채택

---

## 1. 파티셔닝이 필요한 이유

요구사항에서 정의한 대로 단일 서버의 메모리와 디스크 용량은 한계가 있다. 파티셔닝은 전체 데이터를 여러 노드에 나눠 저장하여 이 한계를 극복한다.

```
파티셔닝 없이 (단일 서버)        파티셔닝 적용 후 (분산)

┌─────────────┐                ┌────────┐ ┌────────┐ ┌────────┐
│  서버 1개   │                │ Node A │ │ Node B │ │ Node C │
│             │                │ 33%    │ │ 33%    │ │ 34%    │
│ 전체 데이터 │                │ 데이터 │ │ 데이터 │ │ 데이터 │
│ 100%        │                └────────┘ └────────┘ └────────┘
│             │
│ 용량 한계   │                → 노드 추가 시 수평 확장 가능
│ 단일 장애점 │                → 한 노드 장애 시 다른 노드 서비스 유지
└─────────────┘
```

## 2. 파티셔닝 전략 비교 및 선택

### 2.1 단순 해시 방식 (Modular Hashing)

```
담당 노드 = hash(key) % 노드 수

예: 노드 3개일 때
  hash("user:1001") % 3 = 2  → Node C
  hash("user:2345") % 3 = 0  → Node A
```

**문제점: 노드 수가 변경되면 대부분의 키가 재배치된다.**

```
노드 3개 → 4개로 증설 시

  hash("user:1001") % 3 = 2  → Node C  (기존)
  hash("user:1001") % 4 = 1  → Node B  (변경) ← 담당 노드가 바뀜!

전체 키의 약 75%가 새 노드로 재배치
→ 대규모 데이터 이동 발생 → 서비스 지연
```

### 2.2 안정 해시 방식 (Consistent Hashing) ← 채택

노드와 키를 동일한 해시 공간(링)에 배치한다. 키는 링에서 **시계 방향으로 가장 가까운 노드**가 담당한다.

```
노드 3개 → 4개로 증설 시

기존:  전체 키의 약 33%만 재배치
단순:  전체 키의 약 75% 재배치

→ 데이터 이동량이 1/N로 감소
```

**채택 이유:**

| 기준 | 단순 해시 | 안정 해시 |
|------|---------|---------|
| 노드 추가 시 재배치 비용 | 전체의 ~(N-1)/N | 전체의 ~1/N |
| 노드 제거 시 재배치 비용 | 전체의 ~(N-1)/N | 전체의 ~1/N |
| 부하 균등성 | 높음 | 기본은 낮음 → 가상 노드로 보완 |
| 구현 복잡도 | 낮음 | 중간 |

## 3. 안정 해시 상세 설계

### 3.1 해시 공간과 해시 링

해시 함수로 MurmurHash3를 사용한다. 출력 범위는 0 ~ 2³²-1 (약 43억)이며, 이를 원형 링으로 구성한다.

```
해시 링 (0 ~ 2³²-1 을 원으로 표현)

                    0
                   ───
          270     / N \     90
               A │     │ B
                  \   /
                   ───
                   180

  링 위의 위치:
    Node A: hash("node-a") = 715,827,882  (전체의 약 16.7%)
    Node B: hash("node-b") = 1,431,655,765 (전체의 약 33.3%)
    Node C: hash("node-c") = 2,863,311,531 (전체의 약 66.7%)
```

### 3.2 키 배치 규칙

키를 해시한 값이 링 위에서 어느 위치에 오는지 계산한 뒤, **시계 방향으로 가장 먼저 만나는 노드**가 담당한다.

```
예시: 노드 A(16.7%), B(33.3%), C(66.7%) 위치

  hash("user:1001") = 25%  → 시계 방향 탐색 → Node B 담당
  hash("user:2345") = 10%  → 시계 방향 탐색 → Node A 담당
  hash("session:x") = 80%  → 시계 방향 탐색 → Node A 담당 (링을 한 바퀴 돌아 A)

  담당 범위:
    Node A: (66.7% ~ 16.7%]  ← 링을 넘어가는 구간 포함
    Node B: (16.7% ~ 33.3%]
    Node C: (33.3% ~ 66.7%]
```

### 3.3 노드 추가 시 동작

```
기존: Node A, B, C 운영 중
신규: Node D를 해시 링의 50% 위치에 추가

추가 전:
  Node C 담당: (33.3% ~ 66.7%]

추가 후:
  Node C 담당: (33.3% ~ 50.0%]  ← 범위 축소
  Node D 담당: (50.0% ~ 66.7%]  ← 신규 담당

재배치 대상:
  기존 Node C가 보유한 키 중 (50% ~ 66.7%] 구간의 키만 Node D로 이동
  나머지 노드(A, B)는 영향 없음
```

### 3.4 노드 제거 시 동작

```
기존: Node A, B, C, D 운영 중
Node B 장애 발생 또는 제거

제거 전:
  Node B 담당: (16.7% ~ 33.3%]

제거 후:
  Node C가 Node B의 구간을 흡수
  Node C 담당: (16.7% ~ 50.0%]

재배치 대상:
  Node B가 보유하던 키가 Node C로 이동
  나머지 노드(A, D)는 영향 없음
```

## 4. 가상 노드 (Virtual Node) 설계

### 4.1 가상 노드가 필요한 이유

기본 안정 해시는 노드 수가 적을 때 부하가 불균등하게 분산될 수 있다.

```
물리 노드 3개의 실제 배치 예 (운이 나쁜 경우)

  Node A: 5% 위치
  Node B: 8% 위치  ← A와 B가 너무 가까움
  Node C: 50% 위치

  담당 범위:
    Node A: (50% ~ 5%]  = 55%  ← 데이터 55% 담당
    Node B: (5%  ~ 8%]  = 3%   ← 데이터 3% 담당 (극히 적음)
    Node C: (8%  ~ 50%] = 42%  ← 데이터 42% 담당

→ Node A에 부하가 집중, Node B는 낭비
```

### 4.2 가상 노드 동작 방식

물리 노드 하나가 링 위에 여러 개의 가상 위치(Virtual Node)를 갖는다.

```
가상 노드 수 = 물리 노드당 V개 (본 설계: V = 150)

물리 노드 3개 → 링 위의 가상 노드 450개

  Node A의 가상 노드:
    hash("node-a#0")  = 2.1%
    hash("node-a#1")  = 14.3%
    hash("node-a#2")  = 29.8%
    ... (총 150개)

  Node B의 가상 노드:
    hash("node-b#0")  = 5.7%
    hash("node-b#1")  = 18.2%
    hash("node-b#2")  = 35.1%
    ... (총 150개)

  Node C의 가상 노드:
    hash("node-c#0")  = 1.3%
    hash("node-c#1")  = 9.6%
    hash("node-c#2")  = 22.4%
    ... (총 150개)
```

가상 노드가 링 전체에 고르게 퍼지면서, 각 물리 노드가 담당하는 실제 데이터 비율이 균등해진다.

```
가상 노드 적용 후 부하 분산 (V=150 기준)

  Node A: 전체의 약 33.3% ± 2%
  Node B: 전체의 약 33.3% ± 2%
  Node C: 전체의 약 33.3% ± 2%
```

### 4.3 가상 노드 수(V) 결정 기준

| V 값 | 부하 균등성 | 메모리 사용 | 노드 추가 복잡도 |
|------|-----------|-----------|--------------|
| 10 | 낮음 (편차 ±15%) | 매우 낮음 | 낮음 |
| 50 | 중간 (편차 ±8%) | 낮음 | 중간 |
| **150** | **높음 (편차 ±2%)** | **중간** | **중간** |
| 500 | 매우 높음 (편차 ±1%) | 높음 | 높음 |

**V = 150 채택 이유:**
- 편차 ±2% 수준의 충분한 균등성 확보
- 노드당 메모리 오버헤드 허용 범위 이내
- 노드 추가/제거 시 재배치 계산 비용이 합리적

### 4.4 가상 노드 데이터 구조

코디네이터는 링 정보를 정렬된 배열로 메모리에 유지한다.

```
링 테이블 (가상 노드 목록, 해시값 기준 정렬)

  index  hash값        물리 노드
  ─────  ──────────    ─────────
    0    13,421,772  → Node C
    1    26,843,545  → Node A
    2    40,265,318  → Node B
    3    53,687,091  → Node C
    4    67,108,864  → Node A
    ...  ...           ...
  449   4,281,944,428 → Node B

키 배치 알고리즘:
  target = hash(key)
  index  = 이진 탐색(링 테이블, target) → O(log V*N)
  node   = 링 테이블[index].물리노드
```

## 5. 노드 추가/제거 절차

### 5.1 노드 추가 절차

```
① 새 노드(Node D) 시작
② Node D가 가십 프로토콜로 클러스터에 참여 알림
③ 코디네이터가 링 테이블에 Node D의 가상 노드 150개 삽입
④ 링 테이블 업데이트: Node D가 담당하게 된 구간 계산
⑤ 해당 구간의 키를 기존 담당 노드에서 Node D로 스트리밍 이전
⑥ 이전 완료 후 코디네이터가 Node D를 정식 서비스 노드로 등록
⑦ 기존 노드의 해당 데이터 삭제

이전 중 요청 처리:
  - 이전이 완료되지 않은 키는 기존 노드에서 계속 서비스
  - 이전 완료 후 새 노드로 전환 (무중단)
```

### 5.2 노드 제거 절차 (계획적 제거)

```
① 제거 대상 노드를 "드레이닝(Draining)" 상태로 표시
② 해당 노드가 담당하던 구간을 인접 노드에 이전
③ 이전 완료 확인
④ 링 테이블에서 해당 노드의 가상 노드 150개 제거
⑤ 가십 프로토콜로 클러스터에 제거 완료 알림
```

### 5.3 노드 장애 시 처리 (비계획적 제거)

```
① 가십 프로토콜이 장애 노드 감지 (heartbeat 미갱신)
② 코디네이터가 해당 노드를 "장애" 상태로 표시
③ 장애 노드가 담당하던 요청을 복제본 노드로 즉시 우회
   (Hinted Handoff — 장애 처리 설계서에서 상세 다룸)
④ 장애 노드 복구 후 데이터 동기화 → 정상 복귀
```

## 6. 핫스팟(Hotspot) 방지 전략

특정 키에 요청이 집중되는 핫스팟 상황을 방지하기 위한 보완 전략이다.

### 6.1 키 접미사 분산 (Key Suffix Sharding)

```
문제: "ranking:global" 키에 초당 수만 건 요청 집중

해결: 키에 무작위 접미사를 붙여 여러 파티션에 분산

  "ranking:global:shard-0"  → Node A
  "ranking:global:shard-1"  → Node B
  "ranking:global:shard-2"  → Node C
  "ranking:global:shard-3"  → Node A
  ... (총 shards 수만큼)

읽기 시: 모든 샤드를 읽어 집계
쓰기 시: 랜덤 접미사 선택하여 저장
```

### 6.2 가상 노드 수 동적 조정

특정 노드에 부하가 집중되면, 해당 노드의 가상 노드 수를 줄이고 다른 노드의 가상 노드 수를 늘려 부하를 재분산한다.

```
Node A CPU 사용률 90% 이상 지속 시

  Node A 가상 노드: 150 → 100개로 축소
  Node B, C 가상 노드: 150 → 175개로 확대

  → Node A의 담당 데이터 약 33% → 22%로 감소
  → Node B, C가 차액을 흡수
```

## 7. 파티셔닝 설계 파라미터 요약

| 파라미터 | 값 | 근거 |
|---------|-----|------|
| 해시 함수 | MurmurHash3 (32-bit) | 빠른 속도, 균등한 분포, 암호화 불필요 |
| 해시 공간 크기 | 0 ~ 2³² - 1 | 충분한 분산 정밀도 |
| 가상 노드 수(V) | 노드당 150개 | 편차 ±2% 균등성 확보 |
| 링 테이블 자료구조 | 정렬된 배열 | 이진 탐색 O(log N) |
| 링 테이블 갱신 방식 | 가십 프로토콜로 전파 | 중앙 집중 없이 자율 갱신 |
| 노드 추가 시 재배치 비율 | 전체의 약 1/N | 안정 해시 특성 |
| 핫스팟 대응 | 키 접미사 샤딩 + 가상 노드 동적 조정 | |

## 8. 다음 단계 연결

파티셔닝은 "어느 노드에 저장할지"를 결정한다. 결정된 노드에서 실제로 어떻게 복제하고 일관성을 유지하는지는 다음 문서에서 다룬다.

| 산출물 | 파티셔닝과의 연결 |
|-------|---------------|
| 복제 전략 설계서 | 파티셔닝으로 결정된 Primary 노드에서 N개 복제본 배치 |
| 일관성 모델 설계서 | 쿼럼(W/R/N)으로 복제본 간 일관성 수준 조정 |
| 장애 처리 설계서 | 장애 노드의 파티션 구간을 인접 노드가 임시 담당 |
