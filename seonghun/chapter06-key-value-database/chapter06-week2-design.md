## 키값 저장소 설계

### 요구사항 정리
- 키 값 쌍 10키로바이트 이하
- 큰 데이터 저장 가능
- 높은 가용성 (장애 발생하더라도 빠른 응답)
- 높은 규모 확장성
- 데이터 일관성 수준 조절 가능
- 정말 낮은 지연시간

### 요구사항 해석
- 메모리가 아닌 디스크 저장 ( 큰 데이터 저장 요구 ) 
- 데이터 자동 분할 기능 ( 키/값 쌍 10키로 이하이지만 큰 데이터 저장 가능 요구)
- 파티션 다수
- 내부 자체 캐시 필요
- 디스크 물리적 쓰기 전 완료 

### 의사결정 및 고민한 것들

#### 디스크 VS 메모리
- 메모리는 큰 데이터 저장 불가
- 디스크는 메모리보다 느린 읽기/쓰기 
- <b>결정사항</b>
	- OS 페이지 캐시에 저장 후 성공 응답
	- 자주 쓰는 데이터 메모리에 캐시

#### 데이터 분산 전략
- 데이터는 여러 노드에 고르게 분산되어 저장되어야한다.
- 특정 노드의 네트워크 이슈 등으로 연결이 끊길 수 있다는것을 염두해야 하기 때문에 한 데이터는 특정 한 노드에만 저장되면 안된다.
- 노드의 개수는 유동적이며 노드 수 변경으로 인한 지연시간 증가는 가능한 최소화 해야한다.
- 위와 같은 상황에서 고민한 키 전략
1. 모듈러 
	- 단순한 구현
	- 노드 추가 시 전체 저장된 데이터 재배치 필요
2. 안정 해시 (일관된 해시)
	- 데이터를 원형의 해시 링에 배치하는 방식
	- 노드 추가시 인접 노드들만 데이터 다시 배치
	- 구현 복잡하다
3. 범위 기반 분할 (키 사전준으로 정렬 후 분할 등)
	- 범위 조회 요구사항이 없음
	- 특정 노드로 트래픽과 데이터 몰리는 현상 발생
	- 노드 추가 삭제가 잦은 이 시스템과 맞지 않다
- 2 결정

#### 데이터 복제 전략
- 데이터 쓰기/수정 시 유저가 설정한 최소한 성공해야하는 노드 수 만큼은 같은 데이터 가지고 있어야한다.
- 만약 특정 데이터의 최신 정보를 저장한 노드의 장애시 다른 노드의 이전 버전 조회를 허용할지 말지 결정해야한다.
- 
- 전략들
1. 리더 + 동기 복제 (리더가 성공해도 복제 노드 데이터 쓰기 실패 시 실패 처리)
2. 리더 + 비동기 복제 (리더만 성공하면 나머지 필수 쓰기 되어야하는 노드들은 배치 등으로 처리)
3. 쓰기/읽기 최소한 만족하는 노드 수 작업 성공 시 성공 처리

- 리더를 정해둘 경우 리더가 장애시 바로 다음 리더를 정하는 로직 필요
- 리더 정하지 않는 케이스는 클라이언트가 요청 보내면 최초로 응답한 노드가 이 요청의 코디네이터가 되어 본일관된 해시 에 맞는 노드들에 쓰기 요청
- 만약 클라이언트에서 처음 요청한 코디네이터가 장애시  유저 서비스단에서 내부 메모리에 모든 노드 상태와 ip 리스트 가지는 가십 프로토콜로 다른 노드를 코디네이터로 사용하며 해결

####  큰 데이터 관련
- 한 키 밸류 쌍은 10KB 가 최대
- 그러나 큰 데이터 저장 가능해야한다
- 유저가 아는 키의 밸류는 실제 분할된 키들의 목록
- 이 형태시 안정 해시 에선 청크된 데이터들이 여러 노드에 분산될 확률이 높다.
####  큰 데이터 분할 전략
- 청크 데이터 분산 X 
	- 청크된 데이터들은 같은 노드에 저장
	- 한 노드가 청크된 데이터들을 모아서 온전한 형태로 반환 가능
	- 안정형 해시의 장점인 균등 분산이 깨지며 핫스팟 발생 가능
	- 정족수 노드만큼 대용량 데이터가 복제되는 문제
- 청크 데이터 분산시
	- 균등한 분산
	- 특정 노드 장애시 파일 못합치는 경우 발생 가능
	- 특정 노드의 청크 데이터가 이전 버전일경우 온전한 파일 생성 불가능

- 청크 데이터는 분산하지 않는다.
- 온전한 파일 생성 불가능한 상황이 너무 치명적이라 판단

### 결정사항 정리
1. 비휘발성 저장 장치 저장 및 실제 디스크가 아닌 페이지 캐시 저장
2. 10KB 이상 데이터 자동 분할 필요
3. 파티션 최소 3개
4. 데이터베이스 시스템의 진입점은 1곳
5. 매 순간 일관성 챙기지 않고 배치 잡 등으로 추후에 일관성 잡는 구조
6. 설정값 상 최소 쓰기 완료되어야하는 파티션 개수들의 페이지 캐시 저장 후 성공 응답 전송
